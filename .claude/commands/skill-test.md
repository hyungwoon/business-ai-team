# 스킬 테스트

스킬의 품질을 데이터 기반으로 검증하는 커맨드.

## 사용법

```
/skill-test [스킬명]              # 단일 스킬 Evaluation 테스트
/skill-test [스킬명] --ab         # 스킬 有 vs 無 A/B 테스트
/skill-test --portfolio           # 전체 스킬 포트폴리오 점검
/skill-test [스킬명] --optimize   # 디스크립션 최적화
```

## 테스트 유형

### 1. Evaluation 테스트 (기본)

하나의 스킬을 **여러 입력으로 동시 실행**하고 채점 기준으로 자동 평가.

**절차:**

1. 대상 스킬의 SKILL.md를 읽어 `skill_type`과 `eval_criteria` 확인
2. `eval_criteria`가 없으면 → 스킬 내용을 분석하여 3-5개 평가 기준 자동 생성 → 사용자 확인
3. 3-5개의 다양한 테스트 입력 생성 (엣지 케이스 포함)
4. 병렬 에이전트로 각 입력에 대해 스킬 실행
5. eval_criteria 기준으로 각 결과물 채점 (0-5점)
6. 결과를 테이블로 정리하여 보고

**채점 기준 작성 원칙:**
- 코드가 아닌 **자연어**로 작성
- 한 번에 **하나의 축**만 최적화 (토큰 사용량 OR 톤 일치도 OR 구조 준수)
- 측정 가능한 기준 포함 (글자 수, 섹션 수, 필수 포함 항목 등)

**결과 형식:**

```
## Evaluation 테스트 결과: [스킬명]
테스트 일시: YYYY-MM-DD
스킬 유형: [Capability / Preference]

| 입력 | 기준1 | 기준2 | 기준3 | 평균 |
|------|-------|-------|-------|------|
| 테스트1 | 4/5 | 3/5 | 5/5 | 4.0 |
| 테스트2 | 5/5 | 4/5 | 4/5 | 4.3 |
| ...  | ...   | ...   | ...   | ...  |

총 평균: X.X/5
개선 제안: [구체적 개선 포인트]
```

### 2. A/B 테스트 (--ab)

스킬 적용 결과 vs 스킬 없이 기본 Claude 결과를 **블라인드 비교**.

**절차:**

1. 동일한 입력 3개 생성
2. 각 입력에 대해:
   - **A**: 스킬 적용하여 실행
   - **B**: 스킬 없이 기본 Claude로 실행
3. 심사 에이전트가 A/B를 **모르는 상태**에서 품질 평가
4. 어느 쪽이 더 나은지 판정

**주요 용도:**
- Capability 유형 스킬의 **필요성 재평가** (모델 업그레이드 후)
- "이 스킬이 아직 필요한가?"를 데이터로 판단

**결과 형식:**

```
## A/B 테스트 결과: [스킬명]
스킬 유형: Capability

| 입력 | 승자 | 품질 차이 | 비고 |
|------|------|----------|------|
| 테스트1 | 스킬 有 | +1.2점 | 구조가 더 명확 |
| 테스트2 | 스킬 無 | +0.3점 | 모델이 충분히 잘 함 |
| 테스트3 | 스킬 有 | +0.8점 | 톤 일관성 우수 |

종합 판정: 스킬 유지 / 스킬 제거 권장 / 스킬 수정 필요
```

### 3. 포트폴리오 점검 (--portfolio)

전체 스킬 상태를 한눈에 진단.

**절차:**

1. `.claude/skills/` + `plugins/` 전체 스캔
2. 각 스킬의 디스크립션 길이, 유형, 마지막 테스트 일자 확인
3. 예산 초과 위험, 미분류 스킬, 장기 미테스트 스킬 식별

**결과 형식:**

```
## 스킬 포트폴리오 점검
총 스킬: [N]개 (권장 20 이내)
디스크립션 예산: [N]자 / 추정 한도 [N]자

경고:
- 250자 초과: [스킬 목록]
- 유형 미분류: [스킬 목록]
- 90일+ 미테스트: [스킬 목록]

Capability 스킬 ([N]개): 최근 A/B 테스트 필요 여부
Preference 스킬 ([N]개): 충실도 테스트 권장
```

### 4. 디스크립션 최적화 (--optimize)

스킬의 트리거 정확도를 테스트하고 최적화된 디스크립션 제안.

**절차:**

1. 현재 디스크립션 분석
2. 이 스킬이 발동해야 할 요청 5개 + 발동하면 안 되는 요청 5개 생성
3. 현재 디스크립션으로 트리거 정확도 평가
4. 개선된 디스크립션 제안 (250자 이내, 트리거 키워드 앞쪽 배치)

## SKILL.md 프론트매터 확장

기존 프론트매터에 다음 필드를 **선택적**으로 추가할 수 있다:

```yaml
---
name: content-creation
description: "마케팅 콘텐츠 제작..."
when_to_use: "Use when..."
allowed-tools: [Read, Glob, Grep, WebSearch, WebFetch]
# === 테스트 프레임워크 확장 (선택) ===
skill_type: preference          # capability | preference
eval_criteria:                  # Evaluation 테스트 채점 기준
  - "브랜드 보이스 가이드라인 준수 여부"
  - "CTA 포함 여부"
  - "타겟 고객 페르소나에 맞는 톤"
last_tested: "2026-04-07"       # 마지막 테스트 일자
test_result: 4.2                # 마지막 평균 점수
---
```

**필드가 없어도 테스트는 가능**: 없으면 스킬 내용을 분석하여 자동 생성 후 사용자 확인.

## 테스트 결과 저장

테스트 결과는 `knowledge/_skill-tests.md`에 누적 기록:

```markdown
# 스킬 테스트 기록

| 날짜 | 스킬 | 테스트 유형 | 점수 | 판정 | 비고 |
|------|------|-----------|------|------|------|
| 2026-04-07 | content-creation | eval | 4.2/5 | PASS | 톤 일관성 우수 |
| 2026-04-07 | browse | ab | 스킬有 승 | 유지 | +1.5점 차이 |
```

## 통합 지점

- **skill-budget.md**: 예산 제약 + 유형 분류 기준 참조
- **feedback-learning.md**: 테스트 결과를 knowledge에 학습 항목으로 저장
- **session-closing.md**: 세션 중 스킬 트리거 실패 감지 시 `/skill-test --optimize` 제안
- **eng-workflow.md**: `/health`에서 포트폴리오 점검 연동
