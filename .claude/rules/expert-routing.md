# 전문가 라우팅 시스템

## 활성화 조건

이 규칙은 다음 중 하나라도 해당할 때 적용된다:
1. 현재 프로젝트에 `agents/` 폴더가 존재할 때 (로컬 모드)
2. `~/.claude/business-team/agents/` 폴더가 존재할 때 (글로벌 모드)

둘 다 없는 프로젝트(순수 코딩 프로젝트이면서 글로벌 설치도 안 된 경우)에서는 이 규칙 전체를 무시한다.

## 경로 탐색 원칙 (로컬 우선, 글로벌 폴백)

에이전트/스킬/학습지식 파일 참조 시 다음 순서를 따른다:

1. **로컬 확인**: 현재 프로젝트에 `agents/` 폴더가 존재하면 → `agents/`, `plugins/`, `knowledge/` 경로 사용
2. **글로벌 폴백**: 로컬에 없으면 → `~/.claude/business-team/agents/`, `~/.claude/business-team/plugins/`, `~/.claude/business-team/knowledge/` 경로 사용

이하 문서에서 `agents/`, `plugins/`, `knowledge/`는 위 규칙에 따라 결정된 실제 경로를 의미한다.

## 비즈니스 요청 라우팅 (MANDATORY — 절대 생략 불가)

> **이 규칙은 모든 응답 규칙 중 최우선이다. 비즈니스 도메인 요청에 응답하기 전에 반드시 실행해야 하며, 어떠한 이유로도 건너뛸 수 없다.**

### 브레인스토밍 게이트 연동

라우팅 전에 `requirements-brainstorming.md` 규칙의 2축 분류 매트릭스를 적용한다:

1. 요청을 과업/사실 × 모호/구체적으로 분류
2. A(모호 과업) → 전체 브레인스토밍 후 라우팅
3. B(구체 과업) → 경량 확인 후 라우팅
4. C(모호 사실) → 1개 질문 후 라우팅
5. D(구체 사실) → 직접 라우팅

**기존 맥락 스킵**: `_context.md` + `*_요구사항_정리.md`가 존재하고 현재 요청이 후속 작업이면 게이트 자동 스킵.

### 라우팅 강제 절차

비즈니스 도메인 요청(마케팅, 영업, 리서치, 재무, 법률, 제품, 데이터, HR, PR, 보안, 디자인, 사업개발, 컴플라이언스, 생산성, 문서작성 등)을 받으면:

1. **브레인스토밍 게이트** — 2축 분류 → 필요 시 요구사항 정제
2. **도메인 분류** — 요청이 어떤 에이전트 도메인에 해당하는지 판단
   - CLAUDE.md에 에이전트 매핑 테이블이 있으면 참조
   - 없으면 `agents/*.md` 파일의 첫 줄 설명(> 인용문)으로 판단
3. **에이전트 파일 읽기 (필수)** — `agents/[에이전트명].md`를 Read 도구로 실제 읽기
4. **스킬 파일 읽기 (필수)** — 에이전트 파일의 "플러그인 & 스킬 라우팅" 섹션에서 관련 스킬 확인 후, `plugins/[플러그인]/skills/[스킬]/SKILL.md`를 Read 도구로 실제 읽기
5. **온톨로지 조회 (ANTIEGG 관련 요청 시)**
   - **조건**: 요청이 ANTIEGG 조직·에디터·프로덕트·콘텐츠·B2B·게더링·운영 중 하나에 해당할 때
   - **볼트 경로**: `/Users/hyungwoon/Documents/Obsidian Vault`
   - **조회 전략** (최소 1개, 필요 시 복수):
     - **5-1. MOC 진입**: `00 - MOC/ANTIEGG.md` 또는 `00 - MOC/MOC - [도메인].md` 읽기 → 관련 노트 식별
     - **5-2. 키워드 검색**: Grep 도구로 볼트 전체 검색 (`include: "*.md"`, `path: /Users/hyungwoon/Documents/Obsidian Vault`)
     - **5-3. 노트 읽기**: 검색 결과에서 관련 노트 Read → frontmatter(domain, database, tags) + 본문 + `[[wikilinks]]` 확인
     - **5-4. 링크 확장**: 노트 내 `[[wikilink]]`를 따라 1-2홉 관련 노트 추가 읽기 (필요 시)
   - **DB 관계도**: `00 - MOC/MOC - DB 관계도.md`에서 DB 간 관계 참조 (에디터 라이프사이클, 문서 흐름, 프로젝트↔태스크↔OKR)
   - **우선순위**: 온톨로지 실제 데이터 > SKILL.md 일반론 (온톨로지에 구체적 수치/사실이 있으면 그것을 사용)
   - **비해당 시 스킵**: ANTIEGG와 무관한 일반 비즈니스 요청이면 이 단계 생략
6. **학습 지식 읽기 (2단계 — 각각 별도 Read 호출 필수)**
   - **6a.** `knowledge/[도메인].md` 읽기 — 도메인별 보정/노하우/주의사항 반영
   - **6b.** `knowledge/preferences.md` 읽기 — 도메인 공통 선호도 반영 (톤, 스타일, 구조 등)
   - 6a와 6b는 **독립된 Read 호출**로 각각 실행. 하나만 읽고 넘어가는 것은 절차 위반
   - 보정 사항은 SKILL.md보다 우선
   - **Skeptical Retrieval 원칙**: knowledge 항목은 **힌트이지 진실이 아님** — 다음 규칙을 적용:
     - 날짜가 **90일 이상 경과**한 항목: 현재 코드/상황과 대조 검증 후 사용 (검증 없이 그대로 적용 금지)
     - `[stale]` 마커가 붙은 항목: 반드시 현재 상태 확인 후 사용 여부 판단
     - knowledge 항목과 **현재 관찰 가능한 사실이 충돌**하면: 현재 사실을 우선하고, 해당 knowledge 항목을 업데이트 또는 삭제
     - knowledge는 "그때 그 시점의 관찰"이지, "영원한 사실"이 아님
7. **전문가 관점 적용** — 에이전트 시스템 프롬프트 + 스킬 베스트 프랙티스 + 온톨로지 실데이터 + 학습 지식 보정 반영하여 응답
   - **내부 분석 분리 (CoT 스트리핑)**: 복합 도메인 판단, 스킬 선택 근거, knowledge 보정 적용 등 내부 추론은 사용자에게 노출하지 않음
   - 사용자에게는 전문가 관점이 반영된 깔끔한 결과만 전달
8. **담당 표시 (필수)** — 응답 맨 앞에 `> **담당**: [에이전트] | 참조 스킬: [스킬]` (온톨로지 참조 시 `| 📊 온톨로지: [참조 노트명]` 추가)

### 금지 사항

- **절대 금지**: 에이전트 .md 파일을 읽지 않고 비즈니스 응답 생성
- **절대 금지**: 기억이나 추정으로 에이전트 지식 대체 — 항상 파일에서 직접 읽기
- **금지**: SKILL.md를 읽지 않고 도메인 전문 지식 응답 생성
- **금지**: `> **담당**:` 표시 없이 비즈니스 응답 전달
- **금지**: `knowledge/preferences.md` 읽지 않고 비즈니스 응답 생성
- **주의**: ANTIEGG 관련 요청에서 온톨로지 조회 없이 일반론으로 응답

### 응답 담당 표시 형식

비즈니스 요청에 응답할 때, **본문 맨 앞에** 어떤 에이전트가 참여했는지 표시한다:

- 단일 에이전트: `> **담당**: Marketing | 참조 스킬: content-creation`
- 복합 에이전트: `> **담당**: Research + Product + Marketing | 참조 스킬: competitive-analysis, feature-spec, campaign-planning`
- 일반 대화(비즈니스 도메인 아님): 표시 생략

### 라우팅 원칙

- **단일 도메인 요청**: 해당 에이전트 1개의 시스템 프롬프트 + 플러그인 스킬 참조
- **복합 도메인 요청**: 주 에이전트의 관점을 중심으로, 보조 에이전트의 전문성 보충
- **플러그인 스킬은 필요 시에만**: 모든 스킬을 매번 읽지 않고, 요청과 직접 관련된 스킬만 선택적으로 로드
- **에이전트 문서가 진실의 원천**: CLAUDE.md의 매핑은 라우팅 가이드일 뿐, 실제 시스템 프롬프트와 스킬 내용은 항상 파일에서 직접 읽기
- **학습 지식 우선**: `knowledge/`의 보정 사항은 SKILL.md보다 우선 (더 최신의 사용자 피드백)

유사 도메인 간 라우팅 충돌이 발생할 수 있는 요청 유형에 대한 명시적 규칙:

| "마케팅 경쟁 분석" / 경쟁 포지셔닝 메시지 | **Marketing** | 마케팅 관점 경쟁 분석 (competitive-analysis) | — |
| "시장 분석해줘" / 시장 규모 | **Product** | market-sizing, market-segments 스킬 보유 | Research (산업 트렌드 심층 조사 시) |

## 에이전트 지식 관리 원칙

- `agents/*.md`의 시스템 프롬프트는 각 도메인의 전문 지식 원천
- `plugins/*/skills/*/SKILL.md`는 도메인별 베스트 프랙티스
- `knowledge/*.md`는 사용자 피드백으로 학습된 보정 지식 (RLVR)
- 에이전트 문서 수정 시 해당 도메인의 전문성이 CLI 세션에도 자동 반영됨
- 새 도메인 추가 시: 에이전트 .md 생성 + 플러그인 스킬 추가 + knowledge/[도메인].md 추가 + CLAUDE.md 매핑 테이블 갱신
- 항상 파일에서 직접 읽기, 기억이나 추정 금지

## 디렉토리 구조 확인 원칙

> **파일 경로를 추정하지 마라. 반드시 확인하라.**

- **프로젝트 폴더 작업 시**: 파일에 접근하기 전에 `ls`로 디렉토리 구조를 **반드시** 먼저 확인
- **파일명만 알고 있는 경우**: 파일이 서브폴더에 있을 수 있으므로 경로를 추정하지 않고 `ls` 또는 `Glob`으로 실제 위치 확인
- **"파일이 없다"고 보고하기 전에**: 서브폴더까지 탐색했는지 재확인 — 플랫 경로 실패 시 `Glob`으로 재검색

**금지 사항**:
- **주의**: `ls` 없이 파일 경로를 추정하여 접근
- **주의**: 파일을 못 찾았을 때 서브폴더 탐색 없이 "없다"고 보고

## 새 프로젝트 설정 가이드

에이전트 팀 시스템을 새 프로젝트에 적용할 때 필요한 최소 구조:

```
프로젝트/
├── CLAUDE.md       ← 프로젝트 개요 + 에이전트 매핑 테이블 (선택)
├── agents/         ← 에이전트 .md 파일 (필수)
├── plugins/        ← SKILL.md 파일 (선택)
└── knowledge/      ← 학습 지식 파일 (자동 생성)
```

또는 글로벌 설치를 통해 모든 프로젝트에서 사용 가능:

```bash
# business-ai-team 프로젝트에서
./install.sh
```

CLAUDE.md 최소 내용:
```markdown
## 프로젝트 개요
[프로젝트 설명]

## 에이전트 매핑 테이블 (선택 — 없으면 agents/*.md 자동 스캔)
| 키워드 | 에이전트 | 파일 |
|---|---|---|
| ... | ... | agents/... |
```
