# 세션 후 엔트로피 정리 (Post-Session Entropy Cleanup)

> 세션 종료 시 레포지토리 엔트로피를 줄이는 자동 정리 시스템.
> "100번 반복하면 유지보수 불가능한 레포지토리가 되어 에이전트가 계속 혼란스러워진다."

## 배경

에이전트는 겨우 수용 가능한 수준의 코드를 작성하고 엔트로피 감소를 위해 아무것도 하지 않는다:
- 함수 X의 동작을 B로 변경했지만, 모든 문서는 여전히 동작 A를 참조
- 인터페이스를 변경했지만, 일부 호출자는 이전 인터페이스를 사용
- 새 유틸리티를 추가했지만, 기존 유사 유틸리티가 여전히 존재
- 주석은 3세션 전의 동작을 설명

## 실행 시점

**session-closing.md의 0.5단계**로 실행한다 (학습 지식 반영 확인 후, _context.md 업데이트 전).

## 정리 절차

### 1단계: 변경 분석 (Blast Radius Analysis)

```bash
# 이번 세션에서 변경된 파일 목록
git diff --name-only HEAD~{커밋수}

# 변경된 함수/타입 식별
git diff HEAD~{커밋수} -- '*.ts' '*.tsx' '*.js' '*.jsx'
```

### 2단계: 정리 에이전트 스폰

```
Agent(
  subagent_type="oh-my-claudecode:code-simplifier",
  model="sonnet",  // fresh context
  prompt="[정리 지시]"
)
```

정리 에이전트 입력:
- 변경된 파일 목록 + diff
- 프로젝트의 CLAUDE.md / AGENTS.md 경로
- knowledge/ 디렉토리 경로

### 3단계: 6가지 정리 검사

**a. 스테일 주석 (Stale Comments)**
- 변경된 함수/클래스의 주석이 현재 동작과 일치하는가?
- JSDoc의 @param, @returns가 실제 시그니처와 맞는가?
- "이 함수는 X를 한다"의 X가 실제 동작인가?

**b. 미사용 임포트 (Unused Imports)**
- 변경으로 인해 더 이상 사용되지 않는 import가 있는가?
- 제거된 코드가 참조하던 모듈이 남아 있는가?

**c. 문서 정합성 (Documentation Consistency)**
- CLAUDE.md의 프로젝트 설명이 변경사항을 반영하는가?
- AGENTS.md의 에이전트 설명이 현재 기능과 맞는가?
- _context.md의 "현재 상태"가 정확한가?
- README.md(존재 시)가 업데이트되었는가?

**d. 호출자 정합성 (Caller Consistency)**
- 시그니처가 변경된 함수의 모든 호출자가 업데이트되었는가?
- 타입이 변경된 인터페이스의 모든 구현체가 반영되었는가?
- 이름이 변경된 export의 모든 import가 업데이트되었는가?

**e. 지식 유효성 (Knowledge Validity)**
- `knowledge/*.md`의 항목 중 이번 변경으로 무효화된 것이 있는가?
- SKILL.md의 "실무 보정 사항" 중 더 이상 유효하지 않은 것이 있는가?

**f. 중복 코드 (Duplicate Code)**
- 이번 세션에서 추가된 코드 중 기존 유틸리티와 중복되는 것이 있는가?
- 새 헬퍼 함수가 기존 헬퍼와 유사한 기능인가?

### 3.5단계: Knowledge 통합 (autoDream)

> Claude 네이티브 메모리의 autoDream 패턴을 knowledge/에 이식.
> "메모리는 추가되는 것이 아니라, 지속적으로 편집되는 것이다."

knowledge/ 디렉토리에 학습 항목이 존재하면 다음 통합 작업을 수행한다:

**a. 중복 병합 (Deduplication)**
- 같은 도메인 파일 내에서 의미가 동일한 항목 → 하나로 합치고 날짜를 최신으로 갱신
- 예: "마케팅 예산 비율 보정" 항목이 2건이면 → 최신 1건으로 병합

**b. 모순 해소 (Contradiction Resolution)**
- 같은 주제에 대해 상반되는 보정이 존재하면 → 최신 항목만 유지, 구 항목 삭제
- 예: #1 "비율은 50:30:20" vs #3 "비율은 40:40:20" → #3만 유지

**c. Staleness 정리 (90일 규칙)**
- 90일 이상 경과한 항목 중 **이미 SKILL.md에 반영 완료된 것** → 삭제
- SKILL.md 미반영 항목은 90일이 지나도 유지 (단, staleness 마커 추가: `[stale]`)
- 판단 기준: 해당 SKILL.md의 `## 실무 보정 사항`에 동일 내용이 있는지 확인

**d. 상대→절대 변환 (Temporal Normalization)**
- "최근", "요즘", "얼마 전" → 작성 날짜 기준 구체적 표현으로 변환
- "지난달" → "2026-02 기준" 등
- 날짜 컬럼이 이미 있으므로 주로 내용(보정 내용, 노하우) 컬럼의 모호한 시간 표현 대상

**e. 카운트 동기화**
- 병합/삭제 후 `_index.md`의 카운트 테이블을 실제 행 수와 동기화
- `last_cleaned` 날짜를 오늘로 갱신

**통합 결과 보고:**

```
Knowledge 통합 (autoDream):
- [N]건 중복 병합
- [N]건 모순 해소 (최신 유지)
- [N]건 staleness 정리 (SKILL.md 반영 완료분)
- [N]건 시간 표현 정규화
```

변경이 없으면 (항목이 적거나 이미 정리됨) 보고를 생략한다.

### 4단계: 자동 수정 vs 플래그

| 항목 | 행동 | 이유 |
|------|------|------|
| 미사용 임포트 | **자동 수정** | 안전, 부작용 없음 |
| 스테일 JSDoc | **자동 수정** | 시그니처와 맞추는 것은 안전 |
| 간단한 문서 업데이트 | **자동 수정** | 팩트 업데이트는 안전 |
| 호출자 시그니처 불일치 | **플래그** | 의도를 확인해야 함 |
| 중복 코드 | **플래그** | 통합 방법을 결정해야 함 |
| 지식 무효화 | **플래그** | 삭제 vs 업데이트 결정 필요 |

### 5단계: 정리 커밋

자동 수정된 항목은 별도 커밋으로 저장:

```bash
git add [수정된 파일들]
git commit -m "chore: 세션 후 엔트로피 정리

- 미사용 임포트 제거
- 스테일 주석 업데이트
- 문서 동기화"
```

플래그된 항목은 사용자에게 보고:

```
엔트로피 정리 완료

자동 수정:
- [N]개 미사용 임포트 제거
- [N]개 스테일 주석 업데이트
- [N]개 문서 동기화

검토 필요:
- [파일:줄] — 호출자 시그니처 불일치: [설명]
- [파일] — 기존 유틸리티와 중복 가능: [설명]
```

## 정리 범위 제한

- 이번 세션에서 변경된 파일과 그 **직접 영향권**만 정리
- 프로젝트 전체 정리는 하지 않음 (스코프 폭발 방지)
- 영향권: 변경 파일을 import하는 파일 + 변경된 타입을 사용하는 파일

## 통합 지점

- **session-closing.md**: 0.5단계로 삽입 (학습 반영 후, _context.md 전)
- **ai-slop-cleaner**: 기존 정리 스킬과 보완적 — ai-slop-cleaner는 코드 품질, 이 규칙은 정합성
- **sync-docs**: 문서 동기화 부분에서 활용
- **telemetry-rubrics.md**: entropy_delta 점수 기록
- **feedback-learning.md**: knowledge/ 항목 추가는 feedback-learning이, 정리는 이 규칙의 3.5단계(autoDream)가 담당

## 금지 사항

- **절대 금지**: 정리 없이 세션 종료 (session-closing 체크리스트에 포함됨)
- **절대 금지**: 정리 범위를 전체 프로젝트로 확장 (이번 세션 변경분만)
- **절대 금지**: 플래그 항목을 자동 수정 (사용자 판단 필요)
- **절대 금지**: 정리 커밋과 기능 커밋을 합치기 (별도 커밋 필수)
