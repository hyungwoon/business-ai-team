# 요구사항 브레인스토밍 게이트

> 모호한 비즈니스 요청을 라우팅하기 전에 요구사항을 정제하는 게이트 시스템.
> `/route` 커맨드 및 자동 라우팅 시 1단계로 실행된다.

## 2축 분류 매트릭스

요청을 받으면 다음 2축으로 분류한다:

|  | **모호** (목표·범위·타겟 불명확) | **구체적** (실행 가능한 수준) |
|---|---|---|
| **과업** (결과물 필요) | **A. 전체 브레인스토밍** | **B. 경량 브레인스토밍** |
| **사실** (정보 확인) | **C. 1개 질문 후 라우팅** | **D. 직접 라우팅** |

### 분류 기준

- **과업 vs 사실**: "만들어줘/작성해줘/기획해줘" = 과업, "알려줘/확인해줘/뭐야" = 사실
- **모호 vs 구체적**: 목표·타겟·범위·형식 중 2개 이상 불명확하면 = 모호

---

## A. 전체 브레인스토밍 (HARD GATE)

> 모호한 과업 요청. 충분한 정보 수집 없이 라우팅하면 결과물 품질이 떨어진다.

### 절차

**1단계: 맥락 파악 + 첫 질문**

요청을 인지했음을 알리고, 가장 중요한 1개 질문을 던진다.

```
이해했습니다. [요청 요약] 작업이군요.
좋은 결과물을 위해 몇 가지 여쭤볼게요.

[가장 중요한 질문 1개]
```

**2단계: 명확화 질문 (1개씩, 충분할 때까지)**

- 한 번에 1개 질문만 (사용자 부담 최소화)
- 이전 답변을 반영하여 다음 질문 도출
- 횟수 제한 없음 — 충분히 구체화될 때까지 계속

**핵심 확인 영역** (모두 확인할 필요 없음, 요청에 따라 선별):

| 영역 | 확인 사항 | 예시 |
|------|-----------|------|
| 목표/목적 | 이 작업으로 달성하려는 것 | "매출 증대? 브랜드 인지도? 내부 의사결정?" |
| 대상/타겟 | 결과물의 수신자 또는 대상 고객 | "경영진 보고? 투자자? 고객?" |
| 범위/규모 | 결과물의 깊이와 폭 | "1페이지 요약? 상세 보고서? 실행 계획?" |
| 예산 | 가용 예산 또는 자원 제약 | "예산 범위? 투입 가능 인력?" |
| 일정 | 마감일 또는 시간 제약 | "언제까지 필요? 급한 건가요?" |
| 현재 상태 | 기존 자료, 진행 상황 | "이미 진행한 것이 있나요?" |
| 제약 조건 | 반드시 포함/제외할 사항 | "꼭 포함해야 할 내용? 피해야 할 것?" |
| 성공 기준 | 결과물의 품질 기준 | "어떤 상태면 성공인가요?" |

**3단계: 접근 방식 제안 (2-3가지)**

수집된 정보를 바탕으로 2-3가지 접근 방식을 제안한다.

```
정리해보면 [요약], 다음 접근 방식을 제안드립니다:

**방식 1: [이름]**
- 장점: ...
- 단점: ...

**방식 2: [이름]**
- 장점: ...
- 단점: ...

어떤 방향이 맞으실까요? 또는 조합하셔도 됩니다.
```

**4단계: 요구사항 저장 + 라우팅**

승인된 방향으로 요구사항 정리 문서를 저장한 후 전문가 라우팅을 진행한다.

- 저장 경로: `projects/[프로젝트]/YYYY-MM-DD_요구사항_정리.md`
- 프로젝트 폴더가 없으면 자동 생성
- 저장 후 에이전트 라우팅 절차로 진입

**요구사항 정리 템플릿:**

```markdown
# [프로젝트/요청명] 요구사항 정리

## 기본정보
- **도메인**: [마케팅/영업/리서치/...]
- **목표**: [달성하려는 것]
- **대상**: [결과물 수신자 또는 타겟]

## 우선순위 (MoSCoW)
- **Must**: [반드시 포함]
- **Should**: [가능하면 포함]
- **Could**: [여유 있으면]
- **Won't**: [이번에는 제외]

## 범위 및 제약
- **범위**: [깊이/폭]
- **예산**: [있으면]
- **일정**: [마감]
- **제약**: [제한사항]

## 성공 기준
- [기준 1]
- [기준 2]

## 선택된 접근 방식
[승인된 방식 요약]
```

## B. 경량 브레인스토밍

> 구체적인 과업 요청이지만, 1-2개 확인이 필요한 경우.

### 절차

1. 누락된 핵심 정보 1-2개만 빠르게 확인
2. 확인 후 바로 라우팅
3. 요구사항 문서 저장은 선택적 (복잡도에 따라)

---

## C. 1개 질문 후 라우팅

> 모호한 사실 확인 요청. 범위만 좁히면 된다.

### 절차

1. "어떤 맥락에서 궁금하신 건가요?" 또는 범위를 좁히는 1개 질문
2. 답변 후 바로 라우팅

---

## D. 직접 라우팅

> 구체적인 사실 확인 요청. 게이트 없이 바로 라우팅.

---

## 탈출 조건 (즉시 우회)

사용자가 다음 표현을 사용하면 브레인스토밍을 **즉시 중단**하고 현재 정보로 바로 라우팅:

- "바로 해줘", "그냥 해줘", "알아서 해줘"
- "빨리", "급해", "시간 없어"
- "브레인스토밍 스킵", "질문 그만"

중단 시: `> 바로 진행합니다.` 표시 후 라우팅.

---

## 맥락 기반 게이트 스킵 (0.5단계)

다음 조건을 **모두** 만족하면 브레인스토밍 게이트를 **자동 스킵**:

1. `projects/[프로젝트]/_context.md` 존재 (이미 진행 중인 프로젝트)
2. 관련 `*_요구사항_정리.md` 존재 (이전 요구사항이 정리됨)
3. 현재 요청이 이전 요구사항의 연장선 (후속 작업)

스킵 시: `> 기존 요구사항 기반으로 진행합니다. (게이트 스킵)` 표시 후 라우팅.

**주의**: 같은 프로젝트라도 **전혀 다른 방향의 요청**이면 게이트를 다시 실행한다.
