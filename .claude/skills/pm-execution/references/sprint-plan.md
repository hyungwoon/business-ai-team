## 스프린트 계획

팀 역량을 추정하고, 스토리를 선택 및 순서화하며, 리스크를 파악하여 스프린트를 계획합니다.

### 맥락

**$ARGUMENTS**의 스프린트 계획을 돕습니다.

사용자가 파일(백로그, 속도 데이터, 팀 명단, 또는 이전 스프린트 보고서)을 제공하면 먼저 읽으세요.

### 지시사항

1. **팀 역량 추정**:
   - 팀원 수와 가용 여부 (휴가, 회의, 온콜)
   - 과거 속도 (최근 3개 스프린트의 평균 스토리 포인트)
   - 역량 버퍼: 예상치 못한 작업, 버그, 기술 부채를 위해 15~20% 여유
   - 스토리 포인트 또는 이상적인 시간으로 가용 역량 계산

2. **스토리 검토 및 선택**:
   - 우선순위가 높은 백로그에서 먼저 추출 (가장 높은 우선순위 먼저)
   - 각 스토리가 완료 준비 기준(Definition of Ready)을 충족하는지 확인 (명확한 인수 기준, 추정 완료, 블로커 없음)
   - 커밋 전에 정제가 필요한 스토리에 표시
   - 역량이 차면 스토리 추가 중단

3. **의존성 매핑**:
   - 다른 스토리 또는 외부 팀에 의존하는 스토리 파악
   - 의존적인 스토리를 적절히 순서화
   - 외부 의존성과 담당자 표시
   - 크리티컬 패스 파악

4. **리스크 및 완화 방안 파악**:
   - 불확실성이 높거나 복잡한 스토리
   - 지연될 수 있는 외부 의존성
   - 지식 집중 (특정 한 사람만 할 수 있는 작업)
   - 각 리스크에 대한 완화 방안 제안

5. **스프린트 계획 요약 작성**:

   ```
   스프린트 목표: [성공이 어떤 모습인지 한 문장으로]
   기간: [2주 / 1주 등]
   팀 역량: [X 스토리 포인트]
   커밋된 스토리: [Z개 스토리, Y 스토리 포인트]
   버퍼: [남은 역량]

   스토리:
   1. [스토리 제목] — [포인트] — [담당자] — [의존성]
   ...

   리스크:
   - [리스크] → [완화 방안]
   ```

6. **스프린트 목표 정의**: 스프린트의 주요 가치 전달을 담은 명확한 한 문장.

단계별로 생각하세요. 마크다운으로 저장하세요.

### 스프린트 역량 계산 공식

**활용 시점:** 스프린트 시작 전 팀의 실제 가용 역량을 정량화할 때.

```
가용 역량 = (팀원 수 × 스프린트 일수 × 가용률) - 휴가일 - 온콜일
예: (5명 × 10일 × 0.65) - 3일 = 29.5 이상적 인일
```

| 항목 | 계산 방법 | 예시 |
|------|----------|------|
| 총 인일 | 팀원 수 × 스프린트 일수 | 5 × 10 = 50 |
| 비계획 차감 | 회의, 온콜, 인터뷰 (30~35%) | -17 |
| 휴가/부재 | 개인별 합산 | -3 |
| 버퍼 | 가용 인일의 15~20% | -5 |
| **실제 가용 역량** | **위 항목 합산** | **25 인일** |

### 스토리 포인트 추정 가이드 (피보나치)

**활용 시점:** 백로그 항목의 상대적 크기를 팀이 합의할 때.

| 포인트 | 의미 | 기준 |
|--------|------|------|
| 1 | 매우 작음 | 명확한 변경, 1시간 이내 |
| 2 | 작음 | 단순 작업, 반나절 |
| 3 | 보통 | 익숙한 영역, 일반 작업 |
| 5 | 중간 | 복잡도 있음, 1~2일 |
| 8 | 큼 | 다수 컴포넌트, 분할 검토 |
| 13 | 매우 큼 | 높은 불확실성, 분할 권장 |
| 21 | 에픽 수준 | 스프린트 내 불가, 분할 필수 |

> **규칙:** 13 이상은 반드시 분할하세요. 플래닝 포커로 팀 합의를 권장합니다.

### 스프린트 목표 SMART 템플릿

**활용 시점:** 스프린트 핵심 가치를 명확한 한 문장으로 정의할 때.

| SMART 요소 | 질문 | 좋은 예 | 나쁜 예 |
|------------|------|---------|---------|
| Specific | 무엇을 달성? | 결제 오류율 감소 | 결제 개선 |
| Measurable | 어떻게 측정? | 오류율 5%→1% | 더 좋게 |
| Achievable | 스프린트 내 가능? | 버그 3건 수정 | 전체 리팩토링 |
| Relevant | 제품 목표 연결? | 전환율 OKR 연계 | 기술 부채만 |
| Time-bound | 기한 명확? | 2주 스프린트 내 | 조만간 |

### 스프린트 백로그 우선순위 기준

**활용 시점:** 역량 범위 내에서 스토리 선택 순서를 결정할 때.

1. **스프린트 목표 기여도**: 목표에 직접 기여하는 스토리 우선
2. **의존성 순서**: 다른 스토리의 선행 조건인 항목 우선
3. **리스크 수준**: 불확실성이 높은 항목을 초반에 배치
4. **비즈니스 가치**: PO가 판단한 임팩트 순
5. **기술 부채 비율**: 전체 역량의 15~20%를 기술 부채에 할당

### 스프린트 회의 체크리스트

**활용 시점:** 스프린트 계획 회의 진행 시 누락 방지.

| 단계 | 체크 항목 | 담당 |
|------|----------|------|
| 사전 준비 | ☐ 백로그 정제 완료 | PO |
| 사전 준비 | ☐ 이전 스프린트 속도 확인 | SM |
| 사전 준비 | ☐ 팀 가용성 확인 (휴가/온콜) | SM |
| 파트 1 | ☐ 스프린트 목표 합의 | PO + 팀 |
| 파트 1 | ☐ 우선순위 스토리 선택 | 팀 |
| 파트 2 | ☐ 스토리별 작업 분해 | 개발팀 |
| 파트 2 | ☐ 의존성 및 리스크 식별 | 팀 |
| 마무리 | ☐ 커밋 확정 및 버퍼 확인 | 팀 |
| 마무리 | ☐ 스프린트 보드 업데이트 | SM |
