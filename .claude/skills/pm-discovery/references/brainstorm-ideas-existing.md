## 제품 아이디어 도출 (기존 제품)

지속적 프로덕트 디스커버리를 위한 다각적 아이디에이션. PM, 디자이너, 엔지니어 시각에서 아이디어를 생성하고 최상위 5개를 우선순위화합니다.

### 맥락

**$ARGUMENTS**에 대한 지속적 프로덕트 디스커버리를 수행하는 프로덕트 트리오를 지원합니다.

사용자가 파일(리서치 데이터, 기회 트리, 페르소나)을 제공하면 먼저 읽습니다. 제품 URL을 언급하면 웹 검색을 사용하여 제품을 이해합니다.

### 도메인 맥락

**프로덕트 트리오** (Teresa Torres, *Continuous Discovery Habits*): PM + 디자이너 + 엔지니어가 함께 디스커버리를 수행합니다. "최고의 아이디어는 종종 엔지니어에게서 나옵니다." 디스커버리는 선형적이지 않습니다 — 실험이 실패하면 되돌아갑니다. **기회-솔루션 트리(Opportunity Solution Tree)**(Teresa Torres)를 사용하여 기회 → 솔루션 → 실험을 매핑합니다.

### 지시 사항

사용자가 목표, 목표 세그먼트, 원하는 결과를 설명합니다. 다음 단계를 거칩니다:

1. **기회 파악**: 제품, 목표, 시장 세그먼트, 원하는 결과를 확인합니다. 모호한 부분이 있으면 명확화를 요청합니다.

2. **세 가지 시각에서 아이디에이션** — 각각 5개의 아이디어를 생성합니다:
   - **프로덕트 매니저**: 비즈니스 가치, 전략적 정렬, 고객 영향에 집중
   - **프로덕트 디자이너**: 사용자 경험, 사용성, 즐거움에 집중
   - **소프트웨어 엔지니어**: 기술적 가능성, 데이터 활용, 확장 가능한 솔루션에 집중

3. **상위 5개 아이디어 우선순위 결정** 기준:
   - 명시된 목표와의 전략적 정렬
   - 원하는 결과에 대한 잠재적 영향
   - 실행 가능성과 필요한 노력
   - 기존 솔루션과의 차별성

4. **각 우선순위 아이디어에 대해** 다음을 제공합니다:
   - 명확한 이름과 한 문장 설명
   - 선정된 이유 (근거)
   - 검증해야 할 핵심 가정

단계별로 생각합니다. 아이디어를 명확하고 구조화된 형식으로 제시합니다.

출력이 상당하면 사용자의 작업 공간에 마크다운 문서로 저장합니다.

## SCAMPER 프레임워크

기존 제품의 각 요소를 7가지 관점에서 변형하여 체계적으로 아이디어를 도출합니다:

| 기법 | 질문 | 적용 예시 |
|------|------|----------|
| **S (Substitute)** 대체 | 어떤 요소를 다른 것으로 대체할 수 있는가? | 수동 입력 → AI 자동 완성 |
| **C (Combine)** 결합 | 두 기능을 합쳐 새로운 가치를 만들 수 있는가? | 채팅 + 프로젝트 관리 통합 |
| **A (Adapt)** 적용 | 다른 산업/제품의 접근법을 차용할 수 있는가? | 게이미피케이션 요소 도입 |
| **M (Modify)** 변형 | 크기, 형태, 프로세스를 변경하면 어떻게 되는가? | 대시보드를 모바일 우선으로 재설계 |
| **P (Put to other use)** 다른 용도 | 기존 기능을 다른 목적으로 활용할 수 있는가? | 내부 도구를 고객용으로 공개 |
| **E (Eliminate)** 제거 | 어떤 기능/단계를 제거해도 되는가? | 3단계 가입을 1단계로 축소 |
| **R (Reverse)** 역전 | 순서나 역할을 뒤집으면 어떻게 되는가? | 고객이 가격을 제시 (역경매) |

활용 시점: 기존 제품 개선 아이디어가 고갈되었을 때, 팀이 동일한 관점에 갇혀 있을 때. 각 기법당 2-3개 아이디어를 강제로 생성하면 최소 14개 이상의 후보를 확보할 수 있습니다.

## Reverse Brainstorming (역발상 브레인스토밍)

"어떻게 하면 이 문제를 더 악화시킬 수 있을까?"를 먼저 묻고, 그 반대에서 해결책을 도출합니다:

### 진행 단계

1. **문제 정의**: 해결하려는 문제를 명확히 진술
2. **역전**: "이 문제를 최대한 악화시키려면?" 질문으로 전환
3. **악화 아이디어 수집**: 판단 없이 5-10개 나열 (예: "로딩을 10초로 늘린다")
4. **반전**: 각 악화 아이디어의 정반대를 솔루션으로 변환
5. **평가**: 반전된 아이디어 중 실행 가능한 것을 선별

활용 시점: 일반 브레인스토밍에서 아이디어가 안 나올 때, 팀이 기존 사고 패턴에서 벗어나야 할 때.

## 아이디어 평가 매트릭스

도출된 아이디어를 Impact × Feasibility로 빠르게 분류합니다:

| | 높은 실행 가능성 | 낮은 실행 가능성 |
|---|---|---|
| **높은 임팩트** | **즉시 실행** — 최우선 추진 대상 | **전략적 투자** — 로드맵에 포함, 단계적 접근 |
| **낮은 임팩트** | **빈 틈 메우기** — 여유 리소스로 처리 | **보류** — 현재 시점에서 투자 가치 없음 |

### 평가 기준 상세

| 기준 | 높음 (3점) | 중간 (2점) | 낮음 (1점) |
|------|-----------|-----------|-----------|
| **임팩트** | 핵심 지표 20%+ 개선 | 보조 지표 개선 | 체감 변화 미미 |
| **실행 가능성** | 1-2 스프린트 내 완료 | 1-2개월 소요 | 3개월+ 또는 외부 의존성 |
| **전략 정합성** | 현재 OKR 직접 기여 | 간접적 기여 | OKR과 무관 |

## 브레인스토밍 규칙

효과적인 아이디에이션 세션을 위한 기본 원칙:

1. **양 우선**: 아이디어의 질보다 양을 먼저 추구 — 최소 15개 이상 도출 후 평가
2. **판단 유보**: 아이디어 생성 단계에서 비판 금지. "그건 안 돼"는 평가 단계에서만
3. **결합 환영**: 타인의 아이디어에 덧붙이고 발전시키기 ("Yes, and..." 방식)
4. **시간 제한**: 각 기법당 10-15분. 시간 압박이 창의성을 자극
5. **역할 분리**: 진행자(촉진) ≠ 참여자(아이디어 생성) ≠ 기록자(문서화)
6. **다양한 시각**: 프로덕트 트리오(PM + 디자이너 + 엔지니어) 최소 참여 보장
