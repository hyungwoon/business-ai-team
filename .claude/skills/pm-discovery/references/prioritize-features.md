## 기능 백로그 우선순위화

기능 아이디어 백로그를 평가하고 순위를 매겨 추진할 상위 5개를 선정합니다.

### 컨텍스트

**$ARGUMENTS**의 기능 우선순위화를 지원합니다.

사용자가 파일(스프레드시트, 백로그, 기회 평가서 등)을 제공하면 직접 읽고 분석하세요.

### 도메인 컨텍스트

프레임워크 선택 가이드는 `prioritization-frameworks` 스킬을 참고하세요. 핵심 권장 사항:

**Opportunity Score** (Dan Olsen, *The Lean Product Playbook*)는 고객 문제 평가에 권장됩니다: Opportunity Score = Importance × (1 − Satisfaction), 0–1로 정규화. Importance 높고 Satisfaction 낮을수록 최고의 기회. **문제(기회)**를 우선순위화하세요, 솔루션이 아니라.

**ICE**는 이니셔티브의 빠른 점수 산정에 권장됩니다: Impact (Opportunity Score × # Customers) × Confidence × Ease. **RICE**는 대형 팀을 위해 Reach를 별도 요소로 추가합니다.

### 지시문

사용자가 제품 목표, 원하는 결과, 기능 아이디어를 설명합니다. 다음 단계를 순서대로 수행하세요:

1. **우선순위 파악**: 제품 목표와 성공 지표를 확인합니다.

2. **각 기능을 다음 기준으로 평가**합니다:
   - **Impact(영향도)**: 원하는 결과에 얼마나 기여하는가? 고객 데이터가 있다면 Opportunity Score를 고려하세요.
   - **Effort(노력)**: 개발, 디자인, 조율에 얼마나 리소스가 필요한가?
   - **Risk(리스크)**: 불확실성이 얼마나 존재하는가? 검증이 필요한 가정은 무엇인가?
   - **전략적 정합성**: 제품 비전 및 현재 목표와 얼마나 잘 맞는가?

3. **상위 5개 기능을 추천**합니다:
   - 명확한 순위(1–5위)
   - 각 선정에 대한 간략한 근거
   - 고려된 주요 트레이드오프
   - 우선순위에서 제외된 항목과 그 이유

4. **우선순위 테이블 형태로 제시**합니다(유용한 경우).

단계적으로 생각하세요. 산출물이 방대한 경우 마크다운으로 저장하세요.

## RICE 스코어링 템플릿

정량적 우선순위 산정을 위한 표준 RICE 테이블. RICE = (Reach × Impact × Confidence) / Effort

| 기능 | Reach (분기당 사용자 수) | Impact (0.25-3) | Confidence (50-100%) | Effort (인월) | RICE 점수 |
|------|----------------------|-----------------|---------------------|--------------|----------|
| 기능 A | [예: 5,000] | [예: 2] | [예: 80%] | [예: 3] | [계산] |
| 기능 B | | | | | |
| 기능 C | | | | | |

### Impact 척도 기준
- **3 (Massive)**: 핵심 지표 30%+ 개선 또는 전환율 대폭 상승
- **2 (High)**: 의미 있는 지표 개선, 고객 만족도 뚜렷한 향상
- **1 (Medium)**: 점진적 개선, 일부 사용자에게 유의미
- **0.5 (Low)**: 미미한 개선, nice-to-have 수준
- **0.25 (Minimal)**: 거의 차이 없음, 간접적 영향만

활용 시점: 백로그에 10개 이상의 기능이 있고 데이터 기반 의사결정이 필요할 때. Reach와 Impact 추정에 사용 가능한 데이터(사용자 수, 전환율 등)가 있을수록 효과적.

## Weighted Scoring Model

RICE보다 유연한 다기준 평가. 팀이 중요하게 보는 기준에 가중치를 직접 설정합니다:

| 기능 | 전략 정합성 (×3) | 고객 영향 (×3) | 매출 기여 (×2) | 구현 용이성 (×1) | 리스크 (×1) | 가중 합계 |
|------|----------------|---------------|---------------|----------------|------------|----------|
| 기능 A | [1-5] | [1-5] | [1-5] | [1-5] | [1-5] | [계산] |
| 기능 B | | | | | | |

활용 시점: 전략적 요소(미션 정합성, 기술 부채 등)가 단순 임팩트보다 중요할 때. 이해관계자 간 기준의 중요도가 다를 때 가중치 조정으로 합의 도출.

## MoSCoW 빠른 참조

스코프 결정이 필요할 때 빠르게 분류:

- **Must have**: 이것 없이는 출시 불가. 법적 요구사항, 핵심 사용자 플로우
- **Should have**: 중요하지만 없어도 출시 가능. 차기 스프린트로 이동 가능
- **Could have**: 있으면 좋지만 없어도 됨. 여유 리소스가 있을 때만
- **Won't have**: 이번에는 하지 않음. 명시적 제외로 범위 확정

활용 시점: RICE 점수보다 빠른 의사결정이 필요할 때. 특히 릴리스 범위 조정 회의에서 이해관계자와 트레이드오프를 논의할 때 효과적.

## 실무 팁

- 우선순위 회의 전에 개별 스코어링을 먼저 수행하면 앵커링 편향을 줄일 수 있음
- 분기 1회 이상 우선순위를 재평가 — 시장 상황과 고객 피드백은 빠르게 변함
- "긴급" vs "중요"를 혼동하지 말 것. 긴급한 요청이 항상 높은 Impact는 아님
- 우선순위 결과를 공개적으로 공유해야 "왜 우리 기능은 안 하나요?"에 대한 사전 답변이 됨
