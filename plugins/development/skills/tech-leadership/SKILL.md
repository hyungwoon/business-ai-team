# Tech Leadership Skill

Frameworks for technology strategy, system architecture, engineering management, and technical decision-making.

## 기술 전략 수립

### 기술 비전 문서 구조

```
기술 비전 문서 (Tech Vision Document)
작성 주기: 연 1회 (연초) + 분기별 리뷰

1. 현재 상태 진단
   - 현재 아키텍처 개요 (다이어그램)
   - 주요 기술 부채 목록
   - 팀 역량 맵
   - 현재 시스템 성능/가용성 지표

2. 목표 상태 (12-18개월)
   - 목표 아키텍처
   - 성능/확장성 목표 (RPS, 레이턴시 P99, 가용성 %)
   - 팀 구성 목표

3. 로드맵
   - Q1-Q4 기술 이니셔티브 우선순위
   - 각 이니셔티브별 예상 임팩트 + 비용
   - 의존성 맵

4. 리스크 및 가정
   - 기술적 리스크 TOP 5
   - 외부 의존성 (벤더, 오픈소스)
```

### 기술 스택 선택 프레임워크

기술 선택 시 평가 기준:

| 기준 | 가중치 | 평가 요소 |
|---|---|---|
| 팀 역량 | 30% | 현재 팀의 숙련도, 학습 곡선 |
| 커뮤니티/에코시스템 | 20% | GitHub 활성도, 문서 품질, 취업 시장 |
| 성능/확장성 | 20% | 벤치마크, 기존 레퍼런스 |
| 운영 복잡도 | 15% | 모니터링, 디버깅, 배포 용이성 |
| 비용 | 10% | 라이선스, 인프라, 인력 채용 비용 |
| 장기 지속성 | 5% | 벤더 안정성, 오픈소스 거버넌스 |

**Build vs Buy vs Open Source 결정 트리:**
- 핵심 경쟁력 영역 → **Build** (차별화 원천)
- 비핵심 + 성숙한 솔루션 존재 → **Buy** (SaaS/상용)
- 비핵심 + 커스터마이징 필요 → **Open Source**

## 시스템 아키텍처

### 아키텍처 설계 원칙

**신뢰성 (Reliability):**
- 단일 장애점(SPOF) 제거
- Circuit Breaker 패턴으로 연쇄 장애 방지
- Graceful Degradation (핵심 기능 우선 보호)
- 목표: 가용성 99.9% (월 다운타임 43분) ~ 99.99% (4분)

**확장성 (Scalability):**
- 수평 확장 설계 (Stateless 서비스)
- 캐시 전략: Read-through / Write-behind
- 데이터베이스 샤딩 시점 기준: 단일 DB 5000 TPS 초과
- CDN 활용: 정적 자산 + API 엣지 캐싱

**유지보수성 (Maintainability):**
- 마이크로서비스 분리 기준: 팀 단위 (Conway's Law)
- API 버저닝: URL 버저닝 (`/v1/`, `/v2/`) 또는 헤더
- 모놀리스 → 마이크로서비스: 트래픽 급증 또는 팀 5명+ 시점

### 기술 부채 관리

**부채 분류 (Martin Fowler Technical Debt Quadrant):**

```
                    신중한(Deliberate)
                         ↑
무분별한(Reckless) ←————————————→ 현명한(Prudent)
                         ↓
                    부주의한(Inadvertent)
```

- **현명한 + 신중한:** 출시 후 리팩토링 계획 있음 → 허용
- **현명한 + 부주의한:** 더 나은 방법을 나중에 발견 → 학습 기회
- **무분별한 + 신중한:** "지금은 대충" → 경고 신호
- **무분별한 + 부주의한:** 실수로 발생한 나쁜 코드 → 즉시 수정

**기술 부채 스프린트 비율:**
- 전체 엔지니어링 용량의 20-25%를 기술 부채/인프라에 지속 투자
- "Innovation Token": 불확실한 기술 선택 최대 3개 동시 허용

## 엔지니어링 팀 관리

### 1:1 미팅 가이드

**주간 1:1 (30분) 어젠다:**
```
1. 현황 공유 (5분)
   - 이번 주 진행 중인 것
   - 막히는 것/도움 필요한 것

2. 성장 & 피드백 (15분)
   - 최근 잘한 것 1가지 (구체적)
   - 개선 가능한 것 1가지 (행동 가능한 수준)

3. 커리어 & 동기 (5분)
   - 요즘 가장 의미 있다고 느끼는 일은?
   - 다음에 배우고 싶은 것은?

4. 기타 (5분)
   - 팀/조직에 대한 피드백
   - 팀장이 도울 수 있는 것
```

### 엔지니어 레벨 정의 (Leveling)

| 레벨 | 칭호 | 범위 | 핵심 역량 |
|---|---|---|---|
| L1 | Junior Engineer | 개인 태스크 | 지시에 따라 구현, 코드리뷰 참여 |
| L2 | Engineer | 개인 기능 | 독립적 기능 설계+구현, 주도적 PR |
| L3 | Senior Engineer | 팀 내 모듈 | 설계 주도, 주니어 멘토링, 기술 결정 |
| L4 | Staff Engineer | 팀 간 시스템 | 크로스팀 아키텍처, 기술 전략 기여 |
| L5 | Principal Engineer | 조직 전체 | 기술 비전 수립, 채용 기준 정의 |
| M1 | Engineering Manager | 팀 (5-8인) | 인재 관리, 프로세스, 팀 문화 |
| M2 | Senior EM / Director | 팀군 (15-25인) | 조직 설계, 로드맵, 채용 전략 |

### 개발 프로세스 (Agile/Scrum)

**스프린트 체계:**
- 스프린트 길이: 2주 (권장)
- 스프린트 용량 계산: 팀원 수 × 8일 × 0.7 (여유 30%)
- 백로그 리파인먼트: 주 1회 1시간
- 스프린트 리뷰: 스테이크홀더에게 데모
- 레트로스펙티브: 매 스프린트 종료 후 (Start/Stop/Continue)

**코드리뷰 기준:**
```
PR 크기: 400라인 이하 권장 (초과 시 분할 요청)
리뷰 응답 시간: 영업일 기준 1일 이내
승인 기준: 최소 1명 (Senior 이상) 승인
자동화: CI/CD 테스트 통과 필수

리뷰어 체크리스트:
□ 기능이 요구사항 충족하는가?
□ 엣지케이스 처리됐는가?
□ 테스트 커버리지 충분한가?
□ 성능 문제 없는가? (N+1, 루프 내 DB 쿼리)
□ 보안 취약점 없는가?
□ 코드 가독성/유지보수성 양호한가?
```

## 엔지니어링 메트릭 (DORA Metrics)

개발 조직 건강 지표 4가지:

| 지표 | 정의 | Elite | High | Medium | Low |
|---|---|---|---|---|---|
| **배포 빈도** | 프로덕션 배포 횟수 | 일 1회+ | 주 1회 | 월 1회 | 6개월 1회 |
| **변경 리드타임** | 커밋 → 배포 시간 | < 1시간 | < 1일 | 1주-1달 | 1달+ |
| **변경 실패율** | 배포 후 롤백 비율 | < 5% | 5-10% | 10-15% | > 15% |
| **복구 시간 (MTTR)** | 인시던트 → 복구 | < 1시간 | < 1일 | 1일-1주 | 1주+ |

**측정 및 개선 방법:**
- GitHub Actions 또는 Jira로 배포 빈도/리드타임 추적
- PagerDuty/OpsGenie로 MTTR 자동 계산
- 분기별 DORA 지표 팀 공유 + 개선 목표 설정

## 온콜 & 운영

### SLO/SLI/SLA 설계

```
SLI (Service Level Indicator): 측정하는 지표
  - 가용성: (성공 요청 / 전체 요청) × 100
  - 레이턴시: P99 응답 시간
  - 처리량: 초당 요청 수 (RPS)

SLO (Service Level Objective): 내부 목표
  - 가용성 99.9% (월간)
  - P99 레이턴시 < 500ms
  - 오류율 < 0.1%

SLA (Service Level Agreement): 외부 약속 (SLO보다 낮게 설정)
  - 가용성 99.5% (여유분 0.4% 확보)

에러 버짓 (Error Budget):
  월 43,200분 × (1 - 0.999) = 43.2분 다운타임 허용
  → 에러 버짓 소진 속도 모니터링 → 소진 빠르면 신규 기능 개발 중단
```

## Architecture Decision Records (ADR)

아키텍처 결정을 문서화하여 미래의 팀원이 "왜 이렇게 됐는지" 이해할 수 있게 한다.

### ADR 템플릿

```markdown
# ADR-[번호]: [결정 제목]

**날짜:** YYYY-MM-DD
**상태:** Proposed | Accepted | Deprecated | Superseded by ADR-[번호]
**결정자:** [이름 또는 팀]

## 컨텍스트 (Context)
[이 결정이 필요하게 된 배경. 현재 상황, 직면한 문제, 제약 조건을 서술한다.
기술적 사실뿐 아니라 정치적/사회적 맥락도 포함한다.]

## 결정 (Decision)
[우리가 선택한 것. 능동태로 명확하게 서술한다.
예: "우리는 X를 사용하기로 결정했다. 왜냐하면..."]

## 고려한 대안 (Alternatives Considered)
| 대안 | 장점 | 단점 | 탈락 이유 |
|---|---|---|---|
| [대안 A] | [CUSTOMIZE] | [CUSTOMIZE] | [CUSTOMIZE] |
| [대안 B] | [CUSTOMIZE] | [CUSTOMIZE] | [CUSTOMIZE] |

## 결과 (Consequences)
**긍정적:**
- [CUSTOMIZE: 이 결정으로 얻는 것]

**부정적:**
- [CUSTOMIZE: 이 결정으로 감수하는 것]

**중립적:**
- [CUSTOMIZE: 변화하는 것]

## 후속 조치 (Follow-up)
- [ ] [CUSTOMIZE: 이 결정 이후 해야 할 것]
```

**ADR 운영 규칙:**
- 번호는 순차 증가, 절대 삭제하지 않는다
- 결정이 바뀌면 기존 ADR을 "Superseded"로 표시하고 새 ADR 작성
- `/docs/adr/` 폴더에 보관, Git으로 버전 관리
- 주요 아키텍처 변경 PR에는 ADR 링크 필수

## 기술 인터뷰 설계

### 시스템 디자인 인터뷰 루브릭

**평가 영역 (각 25점, 총 100점):**

| 영역 | 평가 기준 | 5점 (탁월) | 3점 (충족) | 1점 (미달) |
|---|---|---|---|---|
| 요구사항 정의 | 기능/비기능 요구사항 도출 | 모호한 부분 능동적으로 질문, 제약 조건 명확화 | 주요 요구사항 파악 | 요구사항 확인 없이 바로 설계 시작 |
| 규모 추정 | 트래픽/데이터 규모 계산 | 합리적 가정 + 계산 과정 설명 | 대략적 규모 파악 | 규모 고려 없음 |
| 고수준 설계 | 컴포넌트 구성 | 명확한 다이어그램, 트레이드오프 설명 | 주요 컴포넌트 식별 | 단일 컴포넌트 설계 |
| 심층 설계 | 병목/엣지케이스 처리 | 스케일링 전략, 장애 시나리오 대응 | 주요 컴포넌트 상세화 | 표면적 설명만 |

**시스템 디자인 인터뷰 진행 구조 (45분):**
```
0-5분:   요구사항 명확화 (면접관과 질의응답)
5-15분:  고수준 설계 (화이트보드/공유 문서)
15-30분: 핵심 컴포넌트 심층 설계
30-40분: 스케일링 + 엣지케이스 논의
40-45분: 질문 및 마무리
```

### 코딩 인터뷰 구조

**평가 기준 (각 20점, 총 100점):**
- **문제 이해:** 엣지케이스 질문, 예시 확인
- **접근법:** 브루트포스 → 최적화 사고 과정 설명
- **구현:** 동작하는 코드, 변수명 명확성
- **테스트:** 직접 테스트케이스 작성 및 검증
- **복잡도 분석:** 시간/공간 복잡도 설명

### 테이크홈 과제 평가 기준

```
평가 항목 (100점):
□ 기능 완성도 (30점): 요구사항 충족 여부
□ 코드 품질 (25점): 가독성, 구조, 네이밍
□ 테스트 (20점): 커버리지, 엣지케이스
□ 문서화 (15점): README, 설계 결정 설명
□ 확장성 고려 (10점): 추가 기능 확장 용이성

제출 기한: 72시간 (주말 포함 시 96시간)
코드 리뷰 세션: 제출 후 30분 화상 미팅으로 결정 이유 설명
```

## Blameless Post-Mortem

인시던트 이후 비난 없이 시스템 개선에 집중하는 회고 프로세스.

### Post-Mortem 템플릿

```markdown
# Post-Mortem: [인시던트 제목]

**날짜:** YYYY-MM-DD
**심각도:** P0 (서비스 전체 중단) | P1 (주요 기능 장애) | P2 (부분 장애)
**영향 시간:** HH:MM ~ HH:MM (총 X시간 Y분)
**작성자:** [이름]
**리뷰어:** [이름]

## 요약 (Executive Summary)
[3-5문장으로 무슨 일이 있었는지, 영향은 무엇이었는지, 어떻게 해결했는지 서술]

## 영향 (Impact)
- 영향받은 사용자 수: [CUSTOMIZE]
- 영향받은 기능: [CUSTOMIZE]
- 비즈니스 영향: [CUSTOMIZE: 매출 손실, SLA 위반 등]

## 타임라인 (Timeline)
| 시각 | 이벤트 |
|---|---|
| HH:MM | [CUSTOMIZE: 최초 이상 감지] |
| HH:MM | [CUSTOMIZE: 알람 발생] |
| HH:MM | [CUSTOMIZE: 온콜 엔지니어 대응 시작] |
| HH:MM | [CUSTOMIZE: 원인 파악] |
| HH:MM | [CUSTOMIZE: 임시 조치 적용] |
| HH:MM | [CUSTOMIZE: 서비스 정상화] |

## 근본 원인 (Root Cause Analysis)
**5 Whys 분석:**
- Why 1: [CUSTOMIZE]
- Why 2: [CUSTOMIZE]
- Why 3: [CUSTOMIZE]
- Why 4: [CUSTOMIZE]
- Why 5: [CUSTOMIZE: 근본 원인]

## 잘한 것 (What Went Well)
- [CUSTOMIZE]

## 개선할 것 (What Could Be Improved)
- [CUSTOMIZE]

## 액션 아이템 (Action Items)
| 항목 | 담당자 | 기한 | 우선순위 |
|---|---|---|---|
| [CUSTOMIZE] | [이름] | YYYY-MM-DD | P0/P1/P2 |

## 후속 리뷰 (Follow-up)
- 액션 아이템 완료 확인: [날짜]
- 유사 인시던트 재발 여부 모니터링 기간: [기간]
```

**Post-Mortem 문화 원칙:**
- 개인을 탓하지 않는다. 시스템과 프로세스를 개선한다
- 인시던트 발생 후 48시간 이내 초안 작성
- 전체 엔지니어링 팀에 공유 (학습 기회)
- 액션 아이템은 다음 스프린트에 반드시 포함

## 마이그레이션 계획 프레임워크

### Strangler Fig 패턴

레거시 시스템을 점진적으로 교체하는 전략. 새 시스템이 레거시를 "감싸며" 성장한다.

```
단계 1: 파사드(Facade) 구축
  - 레거시 앞에 API 게이트웨이/프록시 배치
  - 모든 트래픽이 파사드를 통과하도록 라우팅

단계 2: 기능 단위 이전
  - 가장 독립적인 기능부터 새 시스템으로 이전
  - 파사드에서 해당 기능 트래픽을 새 시스템으로 라우팅
  - 레거시의 해당 기능은 유지 (롤백 대비)

단계 3: 검증 후 레거시 제거
  - 새 시스템 안정화 확인 (2-4주 모니터링)
  - 레거시 해당 기능 비활성화
  - 반복

단계 4: 레거시 완전 제거
  - 모든 기능 이전 완료 후 레거시 종료
```

### 피처 플래그 전략

```
피처 플래그 유형:
- Release Flag: 미완성 기능 숨기기 (배포 후 점진적 활성화)
- Experiment Flag: A/B 테스트
- Ops Flag: 긴급 기능 비활성화 (킬 스위치)
- Permission Flag: 특정 사용자/그룹에만 노출

플래그 관리 원칙:
- 플래그 수명: 최대 2주 (장기 플래그는 기술 부채)
- 플래그 레지스트리 유지 (이름, 목적, 담당자, 만료일)
- 플래그 제거 PR은 기능 PR과 별도로 관리
```

### 롤백 전략

| 배포 방식 | 롤백 방법 | 소요 시간 | 적합한 상황 |
|---|---|---|---|
| Blue-Green | 트래픽 전환 | < 1분 | 스테이트리스 서비스 |
| Canary | 카나리 비율 0%로 | < 5분 | 점진적 배포 |
| Feature Flag | 플래그 비활성화 | < 1분 | 기능 단위 롤백 |
| DB Migration | 역방향 마이그레이션 | 수분-수시간 | 스키마 변경 (사전 준비 필수) |

## 엔지니어링 예산 계획

### 헤드카운트 모델링

```
엔지니어링 팀 규모 계산 공식:

필요 엔지니어 수 = (총 개발 용량 필요량) / (엔지니어 1인 연간 생산성)

엔지니어 1인 연간 생산성:
- 근무일: 240일/년
- 실제 개발 시간: 240 × 0.6 = 144일 (미팅, 리뷰, 온콜 제외)
- 스프린트 용량: 144일 × 8시간 = 1,152시간/년

헤드카운트 계획 시 고려 요소:
- 이직률 (한국 IT 업계 평균 15-20%/년)
- 채용 리드타임 (시니어: 3-6개월, 주니어: 1-2개월)
- 온보딩 기간 (생산성 100% 도달: 3-6개월)
- 팀 규모 최적화: 팀당 5-8명 (Amazon "Two-Pizza Rule")
```

### 인프라 비용 예측

```
비용 카테고리:
1. 컴퓨팅 (EC2/GKE/AKS): 총 인프라 비용의 40-50%
2. 데이터베이스 (RDS/Cloud SQL): 20-30%
3. 네트워크 (데이터 전송, CDN): 10-15%
4. 스토리지 (S3/GCS): 5-10%
5. 모니터링/로깅 (Datadog, CloudWatch): 5-10%
6. 기타 SaaS 도구: 5-10%

비용 최적화 체크리스트:
□ Reserved Instance / Committed Use 할인 적용 (1-3년 약정, 30-60% 절감)
□ Spot/Preemptible 인스턴스 활용 (배치 작업, 70-90% 절감)
□ 미사용 리소스 정기 감사 (월 1회)
□ 스토리지 라이프사이클 정책 (30일 후 IA, 90일 후 Glacier)
□ 데이터 전송 비용 최소화 (같은 리전 내 통신)
```

### Build vs Buy ROI 계산

```
Build 비용:
- 개발 비용 = 엔지니어 수 × 개발 기간 × 월 인건비
- 유지보수 비용 = 연간 개발 비용의 20-30%
- 기회비용 = 다른 기능 개발 포기

Buy 비용:
- 라이선스/구독료 (월/년)
- 통합 개발 비용 (1-4주)
- 벤더 종속 리스크 (전환 비용)

ROI 판단 기준:
- Build가 유리: 핵심 경쟁력, 고도 커스터마이징 필요, 장기 사용
- Buy가 유리: 비핵심 기능, 빠른 출시 필요, 팀 역량 부족
- 손익분기점: Build 총비용 < Buy 3년 총비용
```

## 오픈소스 거버넌스

### 기여 정책

```
오픈소스 기여 가이드라인:

1. 사전 승인 필요한 경우:
   - 회사 코드/알고리즘이 포함된 기여
   - 새로운 오픈소스 프로젝트 생성 (회사 이름으로)
   - 경쟁사 프로젝트에 기여

2. 사전 승인 불필요:
   - 버그 수정 (회사 코드 미포함)
   - 문서 개선
   - 개인 프로젝트 (업무 시간 외, 회사 자원 미사용)

3. 기여 전 체크리스트:
   □ 라이선스 호환성 확인 (GPL vs MIT vs Apache)
   □ 법무팀 CLA(Contributor License Agreement) 검토
   □ 민감 정보 포함 여부 확인
   □ 팀장 승인
```

### 라이선스 컴플라이언스

| 라이선스 | 상업적 사용 | 소스 공개 의무 | 특허 보호 | 주의 사항 |
|---|---|---|---|---|
| MIT | 허용 | 없음 | 없음 | 가장 자유로움 |
| Apache 2.0 | 허용 | 없음 | 있음 | 특허 분쟁 보호 |
| GPL v3 | 허용 | 필수 (파생물) | 있음 | 상용 제품에 주의 |
| LGPL | 허용 | 라이브러리만 | 있음 | 동적 링크 시 허용 |
| AGPL | 허용 | 네트워크 사용도 | 있음 | SaaS에 매우 주의 |

**보안 스캐닝 도구:**
- Dependabot (GitHub): 취약점 자동 감지 + PR 생성
- Snyk: 오픈소스 취약점 + 라이선스 컴플라이언스
- FOSSA: 엔터프라이즈급 라이선스 관리

## 기술 문서 표준

### RFC (Request for Comments) 템플릿

```markdown
# RFC-[번호]: [제목]

**작성자:** [이름]
**날짜:** YYYY-MM-DD
**상태:** Draft | Review | Accepted | Rejected | Withdrawn
**리뷰 마감:** YYYY-MM-DD

## 요약 (Summary)
[1-2문장으로 이 RFC가 제안하는 것]

## 동기 (Motivation)
[왜 이 변경이 필요한가? 현재 문제점은 무엇인가?]

## 상세 설계 (Detailed Design)
[구체적인 구현 방법. 다이어그램, 코드 예시 포함]

## 트레이드오프 (Trade-offs)
**장점:**
- [CUSTOMIZE]

**단점:**
- [CUSTOMIZE]

## 대안 (Alternatives)
[고려했지만 선택하지 않은 방법과 이유]

## 미해결 질문 (Open Questions)
- [ ] [CUSTOMIZE: 아직 결정되지 않은 것]

## 구현 계획 (Implementation Plan)
- [ ] Phase 1: [CUSTOMIZE]
- [ ] Phase 2: [CUSTOMIZE]
```

### 디자인 문서 구조

```
디자인 문서 (Design Doc) 표준 구조:

1. 개요 (Overview) — 1페이지
   - 문제 정의
   - 제안하는 솔루션 요약
   - 성공 지표

2. 배경 (Background) — 0.5페이지
   - 현재 시스템 상태
   - 관련 이전 결정

3. 상세 설계 (Detailed Design) — 2-5페이지
   - 아키텍처 다이어그램
   - 데이터 모델
   - API 설계
   - 시퀀스 다이어그램

4. 고려 사항 (Considerations)
   - 보안
   - 성능
   - 운영 (모니터링, 알람)
   - 마이그레이션

5. 타임라인 (Timeline)
   - 마일스톤별 예상 완료일
```

## 엔지니어링 문화 진단

### 개발자 경험 (DX) 설문

**분기별 익명 설문 (1-5점 척도):**

```
개발 환경:
□ 로컬 개발 환경 설정이 쉽다 (1-5)
□ CI/CD 파이프라인이 빠르고 안정적이다 (1-5)
□ 필요한 도구와 리소스에 쉽게 접근할 수 있다 (1-5)

코드 품질:
□ 코드리뷰가 학습에 도움이 된다 (1-5)
□ 기술 부채가 업무를 방해하지 않는다 (1-5)
□ 테스트 작성이 장려된다 (1-5)

팀 협업:
□ 팀 내 지식 공유가 활발하다 (1-5)
□ 의사결정 과정이 투명하다 (1-5)
□ 실수해도 비난받지 않는다 (1-5)

성장:
□ 기술적으로 성장하고 있다고 느낀다 (1-5)
□ 커리어 방향에 대해 팀장과 충분히 대화한다 (1-5)

주관식:
- 지금 가장 불편한 개발 경험은?
- 팀이 즉시 개선해야 할 것 1가지는?
```

**DX 점수 해석:**
- 4.0+ : 건강한 엔지니어링 문화
- 3.0-3.9 : 개선 필요 영역 식별 후 분기 내 조치
- 3.0 미만 : 긴급 개입 필요 (팀장 1:1 + 구조적 변화)
