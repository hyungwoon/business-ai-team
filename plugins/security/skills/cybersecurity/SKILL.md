# Cybersecurity Skill

Frameworks for information security risk management, policy design, incident response, and security awareness in business contexts.

## 보안 위험 평가 (Security Risk Assessment)

### 자산 분류 (Asset Classification)

| 분류 | 정의 | 예시 | 보호 수준 |
|---|---|---|---|
| **극비 (Top Secret)** | 유출 시 사업 존속 위협 | 핵심 소스코드, M&A 정보 | 암호화 + 접근 로그 필수 |
| **기밀 (Confidential)** | 유출 시 심각한 손실 | 고객 개인정보, 재무 데이터 | 역할 기반 접근 통제 |
| **내부 (Internal)** | 외부 미공개 정보 | 내부 문서, 인사 정보 | 직원 접근 제한 |
| **공개 (Public)** | 외부 공개 가능 | 마케팅 자료, 공개 API | 기본 보호 |

### STRIDE 위협 모델링

시스템/서비스 설계 검토 시 6가지 위협 유형을 체계적으로 분석:

| 위협 | 정의 | 예시 | 대응 |
|---|---|---|---|
| **S**poofing | 신원 위조 | 피싱, 세션 하이재킹 | MFA, 디지털 서명 |
| **T**ampering | 데이터 무결성 훼손 | SQL 인젝션, 파일 변조 | 입력 검증, 해시 검증 |
| **R**epudiation | 행위 부인 | 거래 부인, 로그 삭제 | 감사 로그, 전자서명 |
| **I**nformation Disclosure | 정보 노출 | 민감정보 평문 전송 | 암호화, 마스킹 |
| **D**enial of Service | 서비스 거부 | DDoS, 리소스 고갈 | Rate limiting, WAF |
| **E**levation of Privilege | 권한 상승 | sudo 취약점 악용 | 최소 권한 원칙 |

## 보안 정책 설계

### 핵심 보안 정책 목록

**반드시 수립해야 할 정책:**

1. **정보보안 정책 (Master Policy)** — 보안의 목적, 범위, 원칙 선언
2. **접근 통제 정책** — 역할 기반 접근(RBAC), 최소 권한 원칙
3. **비밀번호 정책** — 복잡도, 만료 주기, MFA 요구사항
4. **암호화 정책** — 저장 및 전송 데이터 암호화 기준
5. **업무용 기기 정책 (AUP)** — 회사 기기 사용 규정, BYOD 가이드
6. **인시던트 대응 정책** — 보고 절차, 대응 팀, 복구 기준
7. **백업 및 복구 정책** — 백업 주기, 보존 기간, RTO/RPO
8. **공급업체 보안 정책** — 벤더 보안 평가, 계약 요구사항
9. **원격 근무 보안 정책** — VPN, 홈 네트워크, 클라우드 사용 규정
10. **퇴직자 보안 처리 절차** — 계정 비활성화, 기기 반납, 데이터 삭제

### 비밀번호/인증 기준 (NIST SP 800-63B 기반)

```
비밀번호 최소 요구사항:
- 최소 길이: 12자 이상
- 복잡도: 대소문자 + 숫자 + 특수문자 조합
- 만료 주기: 90일 (침해 시 즉시 변경)
- 재사용 제한: 최근 10개 비밀번호 재사용 금지
- 잠금: 5회 연속 실패 시 15분 잠금

다단계 인증 (MFA) 의무 대상:
- 모든 클라우드 서비스 관리자 계정
- 원격 접속 (VPN, SSH)
- 고객 데이터 접근 시스템
- CI/CD 파이프라인

권장 MFA 방식 (강도 순):
1. 하드웨어 보안키 (FIDO2/WebAuthn) — 최강
2. TOTP 앱 (Google Authenticator, Authy)
3. Push 알림 (Duo, Microsoft Authenticator)
4. SMS OTP — 피싱 취약, 최후 수단으로만
```

## 인시던트 대응 (Incident Response)

### IR 프로세스 (NIST SP 800-61 기반)

**Phase 1: 준비 (Preparation)**
- IR 팀 구성 및 역할 정의
- IR 도구 준비 (SIEM, EDR, 포렌식 도구)
- 연락처 목록 최신화 (내부 + 외부 — 법무, CERT)
- IR 훈련 및 시뮬레이션 (연 1회 이상)

**Phase 2: 탐지 및 분석 (Detection & Analysis)**

인시던트 심각도 분류:

| 등급 | 기준 | 대응 시간 | 에스컬레이션 |
|---|---|---|---|
| Critical | 랜섬웨어, 데이터 유출 확인 | 즉시 | CISO + CEO + 법무 |
| High | 권한 외 접근, 악성코드 감염 | 1시간 내 | CISO + IT팀장 |
| Medium | 비인가 접근 시도 | 4시간 내 | IT팀장 |
| Low | 의심 활동, 정책 위반 | 24시간 내 | 보안 담당자 |

**Phase 3: 봉쇄 (Containment)**
- 단기 봉쇄: 감염 시스템 네트워크 분리
- 장기 봉쇄: 임시 패치, 접근 통제 강화
- 증거 보존: 포렌식 이미지 확보 전 시스템 수정 금지

**Phase 4: 제거 (Eradication)**
- 악성코드 완전 제거
- 취약점 패치
- 손상된 계정 비활성화 및 재생성

**Phase 5: 복구 (Recovery)**
- 시스템 복원 및 모니터링 강화
- 정상 운영 재개 기준 정의
- 72시간 집중 모니터링

**Phase 6: 사후 학습 (Post-Incident)**
- PIR (Post-Incident Review): 5일 내 실시
- 타임라인 재구성
- 근본 원인 분석 (Root Cause Analysis)
- 재발 방지 조치 이행 계획

## 클라우드 보안 체크리스트 (AWS/GCP/Azure 공통)

### 기본 보안 설정

```
IAM/접근 관리:
[ ] 루트/관리자 계정 MFA 활성화
[ ] 역할 기반 접근 통제 (RBAC) 적용
[ ] 서비스 계정 키 정기 로테이션 (90일)
[ ] 미사용 계정/권한 정기 검토 및 삭제

네트워크:
[ ] 기본 보안 그룹 최소 권한 설정 (0.0.0.0/0 금지)
[ ] VPC/서브넷 분리 (공개/내부/DB)
[ ] 불필요한 공개 IP 제거
[ ] WAF 활성화 (웹 서비스)

데이터:
[ ] 저장 데이터 암호화 (서버 사이드)
[ ] 전송 데이터 TLS 1.2+ 강제
[ ] S3/GCS 버킷 공개 접근 차단
[ ] 백업 암호화 및 별도 계정 보관

모니터링:
[ ] CloudTrail/Audit Log 활성화
[ ] 이상 행동 알림 설정 (GuardDuty, Security Command Center)
[ ] 비용 이상 알림 (크립토재킹 탐지)
[ ] 보안 점수 대시보드 주간 검토
```

| 5 | 개인정보보호 (PIPA) | 온라인 모듈 + 퀴즈 |

- 분기별 1회 모의 피싱 훈련

## Zero Trust 아키텍처

### 핵심 원칙

Zero Trust는 "절대 신뢰하지 말고, 항상 검증하라(Never Trust, Always Verify)"는 보안 모델이다. 내부 네트워크라도 신뢰하지 않는다.

**5대 원칙:**
1. **명시적 검증** — 모든 접근 요청을 ID, 위치, 기기 상태, 서비스 등 모든 데이터 포인트로 검증
2. **최소 권한 접근** — 사용자 접근을 Just-In-Time(JIT), Just-Enough-Access(JEA)로 제한
3. **침해 가정** — 이미 침해됐다고 가정하고 폭발 반경 최소화, 세그먼트 분리, E2E 암호화
4. **마이크로 세그멘테이션** — 네트워크를 소규모 구역으로 분리하여 측면 이동 차단
5. **지속적 모니터링** — 모든 접근 로그 수집 및 이상 행동 실시간 탐지

### Zero Trust 구현 체크리스트

```
ID & 접근 관리:
[ ] 모든 사용자/기기에 강력한 ID 인증 (MFA 필수)
[ ] 조건부 접근 정책 구현 (기기 상태, 위치, 리스크 점수 기반)
[ ] 특권 계정 관리 (PAM) 솔루션 도입
[ ] 서비스 계정 및 API 키 중앙 관리

기기 보안:
[ ] 모든 기기 등록 및 상태 모니터링 (MDM/EMM)
[ ] 기기 컴플라이언스 정책 (OS 패치, 암호화, 안티바이러스)
[ ] 비관리 기기의 기업 리소스 접근 차단 또는 제한

네트워크:
[ ] 마이크로 세그멘테이션 구현
[ ] 내부 트래픽도 암호화 (mTLS)
[ ] DNS 필터링 및 웹 프록시 적용
[ ] VPN 대신 ZTNA(Zero Trust Network Access) 전환 검토

애플리케이션:
[ ] 앱별 접근 정책 (사용자 역할 + 기기 상태 기반)
[ ] API 게이트웨이를 통한 모든 API 접근 통제
[ ] 세션 타임아웃 및 재인증 정책

데이터:
[ ] 데이터 분류 및 레이블링 자동화
[ ] DLP(Data Loss Prevention) 정책 적용
[ ] 암호화 키 관리 중앙화 (KMS)
```

## 공급망 보안 (Supply Chain Security)

### SBOM (Software Bill of Materials)

SBOM은 소프트웨어에 포함된 모든 컴포넌트 목록이다. 미국 행정명령 14028 이후 글로벌 표준으로 자리잡고 있다.

**SBOM 필수 포함 항목:**
- 컴포넌트 이름 및 버전
- 공급업체/제조사
- 라이선스 정보
- 알려진 취약점 (CVE 연계)
- 의존성 관계 (직접/간접)

**SCA(Software Composition Analysis) 도구:**

| 도구 | 유형 | 특징 |
|---|---|---|
| Snyk | 상용 | 개발자 친화적, CI/CD 통합 |
| OWASP Dependency-Check | 오픈소스 | Java/Python/Node 지원 |
| Trivy | 오픈소스 | 컨테이너 이미지 스캔 강점 |
| Black Duck | 상용 | 엔터프라이즈급, 라이선스 관리 |
| Grype | 오픈소스 | 빠른 취약점 스캔 |

### 벤더 보안 평가 설문 템플릿

```
공급업체 보안 평가 질문지 (핵심 항목)

1. 인증 및 컴플라이언스
   [ ] ISO 27001 인증 보유 여부 및 유효기간
   [ ] SOC 2 Type II 보고서 제공 가능 여부
   [ ] 국내: ISMS-P 인증 여부
   [ ] 최근 외부 보안 감사 실시 여부 (1년 이내)

2. 데이터 보호
   [ ] 고객 데이터 저장 위치 (국가/리전)
   [ ] 데이터 암호화 방식 (저장/전송)
   [ ] 데이터 보존 기간 및 삭제 정책
   [ ] 개인정보 처리 방침 및 PIPA 준수 여부

3. 접근 통제
   [ ] MFA 적용 범위
   [ ] 특권 계정 관리 방식
   [ ] 퇴직자 계정 비활성화 절차 및 소요 시간

4. 인시던트 대응
   [ ] 보안 인시던트 고객 통보 SLA (몇 시간 이내?)
   [ ] 최근 3년 내 주요 보안 인시던트 이력
   [ ] 사업 연속성 계획(BCP) 보유 여부

5. 개발 보안
   [ ] SDLC에 보안 검토 포함 여부
   [ ] 정기 침투 테스트 실시 여부 (연 1회 이상)
   [ ] 취약점 패치 SLA (Critical: 며칠 이내?)
```

## 보안 성숙도 모델 (NIST CSF 기반)

### 5개 기능별 성숙도 레벨

NIST Cybersecurity Framework의 Identify, Protect, Detect, Respond, Recover 5개 기능에 대해 레벨 1-5로 평가:

| 레벨 | 명칭 | 정의 |
|---|---|---|
| 1 | Initial | 비공식적, 임시방편적 대응. 문서화 없음 |
| 2 | Developing | 일부 프로세스 존재하나 일관성 없음 |
| 3 | Defined | 공식 정책/절차 수립, 전사 적용 |
| 4 | Managed | 측정 가능한 지표로 관리, 지속적 개선 |
| 5 | Optimizing | 자동화, 예측적 보안, 업계 선도 |

### 기능별 레벨 평가 기준

**Identify (식별):**
```
L1: 자산 목록 없음, 리스크 평가 미실시
L2: 주요 자산 목록 존재, 비정기 리스크 평가
L3: 전체 자산 인벤토리, 연간 리스크 평가, 공급망 리스크 포함
L4: 실시간 자산 관리, 분기별 리스크 평가, 위협 인텔리전스 활용
L5: 자동화된 자산 발견, 지속적 리스크 모니터링, 예측적 위협 분석
```

**Protect (보호):**
```
L1: 기본 방화벽만 존재, MFA 미적용
L2: 일부 시스템 MFA, 패치 관리 비정기적
L3: 전사 MFA, 정기 패치 관리, 보안 교육 연 1회
L4: Zero Trust 부분 적용, 자동화된 패치, 역할별 교육
L5: 완전한 Zero Trust, 자동화된 보안 통제, 지속적 교육
```

**Detect (탐지):**
```
L1: 로그 수집 없음, 수동 탐지
L2: 기본 로그 수집, 알림 일부 설정
L3: SIEM 운영, 24/7 모니터링, 인시던트 분류 체계
L4: EDR/XDR 운영, 행동 분석, 위협 헌팅
L5: AI 기반 이상 탐지, 자동화된 위협 헌팅
```

**Respond (대응):**
```
L1: IR 계획 없음, 임시 대응
L2: 기본 IR 절차 존재, 미훈련
L3: 공식 IR 계획, 연 1회 훈련, 외부 CERT 연계
L4: 자동화된 초기 대응, 분기별 훈련, 위기 커뮤니케이션 계획
L5: 자동화된 봉쇄/격리, 지속적 훈련, 업계 정보 공유
```

**Recover (복구):**
```
L1: 백업 없음 또는 미검증
L2: 정기 백업, 복구 미테스트
L3: 검증된 백업, RTO/RPO 정의, 복구 훈련 연 1회
L4: 자동화된 복구, 분기별 DR 테스트, 비즈니스 연속성 계획
L5: 무중단 복구 목표, 지속적 DR 검증, 탄력적 아키텍처
```

## DLP (Data Loss Prevention) 정책 프레임워크

### 데이터 분류 기반 통제 매트릭스

| 데이터 분류 | 이메일 외부 전송 | USB 저장 | 클라우드 업로드 | 화면 캡처 |
|---|---|---|---|---|
| 극비 | 차단 | 차단 | 차단 | 워터마크 |
| 기밀 | 승인 필요 | 암호화 필수 | 승인된 서비스만 | 허용 (로그) |
| 내부 | 경고 표시 | 허용 (로그) | 허용 (로그) | 허용 |
| 공개 | 허용 | 허용 | 허용 | 허용 |

### DLP 구현 레이어

**엔드포인트 DLP:**
- 기기에서 데이터 이동 모니터링 (USB, 프린터, 클립보드)
- 도구: Microsoft Purview, Symantec DLP, Forcepoint

**네트워크 DLP:**
- 이메일, 웹, FTP 등 네트워크 트래픽 검사
- SSL/TLS 복호화 검사 (프라이버시 정책 고지 필수)

**클라우드 DLP (CASB):**
- SaaS 앱 내 데이터 이동 통제
- 승인되지 않은 클라우드 서비스 차단 (Shadow IT)
- 도구: Microsoft Defender for Cloud Apps, Netskope, Zscaler

**DLP 정책 구현 시 주의사항:**
- 직원 프라이버시 보호 (개인 기기 vs 업무 기기 구분)
- 개인정보보호법(PIPA) 준수 — 모니터링 사실 고지 의무
- 오탐(False Positive) 최소화 — 업무 방해 방지
- 단계적 도입 (모니터링 → 경고 → 차단 순서)

## 버그 바운티 프로그램 설계

### 프로그램 구성 요소

**스코프 정의:**
```
In-Scope:
- [CUSTOMIZE — 웹 애플리케이션 URL]
- [CUSTOMIZE — API 엔드포인트]
- [CUSTOMIZE — 모바일 앱]

Out-of-Scope:
- 소셜 엔지니어링 공격
- 물리적 보안 공격
- 서드파티 서비스 취약점
- 이미 알려진 취약점 (공개 CVE)
- DoS/DDoS
```

**보상 체계 (Reward Structure):**

| 심각도 | CVSS 범위 | 보상 범위 |
|---|---|---|
| Critical | 9.0-10.0 | 3,000,000원 ~ 10,000,000원 |
| High | 7.0-8.9 | 1,000,000원 ~ 3,000,000원 |
| Medium | 4.0-6.9 | 300,000원 ~ 1,000,000원 |
| Low | 0.1-3.9 | 50,000원 ~ 300,000원 |

**법적 세이프 하버 (Legal Safe Harbor) 필수 문구:**
```
[회사명]은 본 버그 바운티 프로그램의 가이드라인을 준수하여
선의로 취약점을 발견하고 보고한 보안 연구자에 대해 법적 조치를
취하지 않을 것을 약속합니다. 단, 다음 조건을 충족해야 합니다:
- 개인정보 또는 기밀 데이터에 접근하지 않을 것
- 서비스 가용성을 저해하지 않을 것
- 발견 즉시 보고하고 공개 전 패치 완료를 기다릴 것
```

**트리아지 프로세스:**
1. 제출 접수 → 자동 확인 이메일 (즉시)
2. 초기 검토 → 유효성 확인 (3영업일 이내)
3. 심각도 평가 → 보상 결정 (7영업일 이내)
4. 수정 완료 → 보상 지급 (패치 후 30일 이내)
5. 공개 허용 → 연구자 크레딧 (합의 후)
