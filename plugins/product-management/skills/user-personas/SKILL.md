# 사용자 페르소나

## 목적
사용자 기반의 진정한 다양성을 포착하는 상세하고 실행 가능한 사용자 페르소나를 리서치 데이터로부터 작성합니다. 이 스킬은 제품 결정을 안내하기 위해 해결 과제(jobs-to-be-done), 페인 포인트, 원하는 결과, 예상치 못한 행동 인사이트를 포함한 리서치 기반 페르소나를 생성합니다.

## 지시문

당신은 페르소나 개발 및 사용자 리서치 종합을 전문으로 하는 경험 많은 제품 리서처입니다.

### 입력
**$ARGUMENTS**에 대한 3개의 정제된 사용자 페르소나를 작성하는 것이 목표입니다.

사용자가 CSV, Excel, 설문 응답, 인터뷰 녹취록, 또는 기타 리서치 데이터 파일을 제공하는 경우 가용 도구를 사용하여 직접 읽고 분석합니다. 핵심 패턴, 인구통계, 동기, 행동을 추출합니다.

### 분석 단계 (단계별로 사고)

1. **데이터 수집**: 제공된 모든 리서치 데이터 및 문서 읽기 및 검토
2. **패턴 인식**: 사용자 전반에 걸친 반복되는 특성, 목표, 페인 포인트, 행동 식별
3. **세분화**: 공유된 동기와 해결 과제를 기반으로 유사한 사용자를 뚜렷한 페르소나로 그룹화
4. **보강**: 각 페르소나에 대해 데이터를 일관된 프로필로 종합
5. **검증**: 페르소나가 실제 리서치 결과에 근거하고 있는지 확인하기 위해 인사이트 교차 검토

### 출력 구조

3개의 각 페르소나에 대해 제공:

**페르소나 이름 & 인구통계**
- 나이대, 역할/직함, 회사 규모 (B2B인 경우), 핵심 특성

**주요 해결 과제(Job-to-be-Done)**
- 페르소나가 달성하려는 핵심 결과
- 과제의 맥락 및 빈도

**상위 3가지 페인 포인트**
- 과제 완수를 방해하는 구체적인 도전 또는 장애물
- 각 페인 포인트의 영향 및 심각도

**상위 3가지 원하는 이점**
- 페르소나가 추구하는 혜택, 결과, 또는 솔루션
- 성공을 측정하는 방법

**한 가지 예상치 못한 인사이트**
- 데이터에서 도출된 직관적이지 않은 행동 패턴 또는 동기
- 제품 결정에 있어 이것이 중요한 이유

**제품 적합도 평가**
- $ARGUMENTS가 이 페르소나의 니즈를 어떻게 해결하는지 (또는 해결할 수 있는지)
- 잠재적 마찰 지점 또는 충족되지 않은 니즈

## 모범 사례

- 모든 인사이트를 실제 데이터에 근거; 가정 지양
- 가능한 경우 리서치의 직접 인용 활용
- 인구통계 범주뿐 아니라 행동 패턴 식별
- 페르소나를 가능한 한 뚜렷하고 비중복적으로 만들기
- 데이터 공백 또는 추가 리서치가 필요한 영역 표시

## 페르소나 카드 템플릿

| 항목 | 내용 |
|------|------|
| **이름/유형** | [가상 이름] — [한 줄 설명] |
| **인구통계** | 나이, 직업, 회사 규모, 기술 숙련도 |
| **목표** | 이 사람이 달성하려는 것 (1-3개) |
| **좌절점** | 현재 겪고 있는 핵심 불만 (1-3개) |
| **행동 패턴** | 제품/서비스 사용 빈도, 방식, 맥락 |
| **대표 인용문** | "실제 인터뷰에서 나온 말" |
| **JTBD** | [상황]일 때, [행동]하고 싶다. 그래야 [기대 결과]하니까. |
| **제품 관계** | 파워 유저 / 일반 사용자 / 이탈 위험 |

활용 시점: 페르소나를 팀 전체가 빠르게 참조할 수 있는 1페이지 형식으로 정리할 때. 벽에 붙이거나 PRD 상단에 첨부.

## Proto-persona vs 리서치 기반 페르소나

| 구분 | Proto-persona | 리서치 기반 페르소나 |
|------|-------------|-------------------|
| **데이터 근거** | 팀의 가정과 경험 | 인터뷰, 설문, 행동 데이터 |
| **작성 시간** | 1-2시간 워크숍 | 2-4주 리서치 사이클 |
| **정확도** | 낮음 — 검증 필요 | 높음 — 데이터 뒷받침 |
| **적합 시점** | 초기 스타트업, MVP 전 | 제품-시장 적합성 이후, 스케일업 |
| **위험** | 확증 편향에 빠지기 쉬움 | 과도한 세분화, 분석 마비 |
| **전환 시점** | 고객 인터뷰 10건 이상 축적 시 리서치 기반으로 전환 |

활용 시점: Proto-persona는 가설 검증의 출발점. "이 가정이 맞는가?"를 테스트하는 도구로 사용하고, 데이터가 쌓이면 리서치 기반 페르소나로 발전시킨다.

## 페르소나 검증 체크리스트

- [ ] 실제 사용자 데이터(인터뷰 5건+, 설문, 행동 로그)에 근거하는가?
- [ ] 각 페르소나가 서로 명확히 구별되는가? (목표, 행동, 페인 포인트 차이)
- [ ] 인구통계가 아닌 행동과 동기 기반으로 세분화했는가?
- [ ] 팀원 3명 이상이 "이 사용자 알겠다"고 공감하는가?
- [ ] 제품 결정에 실제로 영향을 줄 수 있는 수준의 구체성이 있는가?
- [ ] 6개월 이내 데이터로 최신성이 유지되고 있는가?
- [ ] 페르소나에 맞지 않는 반례(counter-example)를 설명할 수 있는가?

## Jobs-to-be-Done 통합

페르소나와 JTBD를 결합하면 "누가"와 "왜"를 동시에 포착합니다:

### JTBD 통합 공식

> **[페르소나 이름]**이 **[상황/맥락]**일 때,
> **[기능적 목표]**를 달성하고 싶어 한다.
> 이를 통해 **[감정적 목표]**를 느끼고,
> **[사회적 목표]**로 인식되기를 원한다.

| JTBD 레이어 | 설명 | 예시 |
|-------------|------|------|
| **기능적** | 실질적으로 완수하려는 과업 | 주간 보고서를 30분 내 완성 |
| **감정적** | 과업 수행 시 느끼고 싶은 감정 | 자신감, 통제감, 안도감 |
| **사회적** | 타인에게 보여지고 싶은 이미지 | 능숙한 전문가, 혁신적인 리더 |

## 안티패턴

페르소나 작성 시 흔히 범하는 실수:

| 안티패턴 | 문제 | 대안 |
|----------|------|------|
| **엘라스틱 페르소나** | "모든 사용자"를 대표하려는 페르소나 | 구체적 세그먼트에 집중 |
| **인구통계 과잉** | 나이/성별/직업만 나열, 행동 부재 | 행동, 동기, 페인 포인트 중심 |
| **셀프 레퍼런싱** | 팀원 자신을 투영한 페르소나 | 실제 사용자 데이터 필수 |
| **냉동 페르소나** | 한 번 만들고 업데이트 없음 | 분기별 검증 및 갱신 |
| **너무 많은 페르소나** | 5개 이상으로 집중력 분산 | 핵심 2-3개에 집중, 나머지는 부차적 |
| **소설 쓰기** | 가상의 스토리가 데이터를 압도 | 모든 인사이트에 데이터 출처 명시 |
