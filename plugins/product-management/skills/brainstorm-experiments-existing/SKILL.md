## 실험 설계 (기존 제품)

전체 구현에 착수하기 전에 제품 가정을 검증하기 위한 저비용 실험을 설계합니다.

### 맥락

**$ARGUMENTS**에 대한 실험을 설계하도록 제품 팀을 지원합니다. 팀에는 기능 아이디어와 검증이 필요한 가정이 있습니다.

사용자가 파일(PRD, 가정 목록, 디자인)을 제공하면 먼저 읽습니다.

### 지시 사항

사용자가 아이디어와 가정을 설명합니다. 다음 단계를 거칩니다:

1. **아이디어와 가정 명확화**: 팀이 무엇을 만들고 싶은지, 무엇을 검증해야 하는지 확인합니다.

2. **각 가정에 대한 실험 제안**. 다음과 같은 방법을 고려합니다:
   - 프로토타입으로 첫 번째 클릭 테스트 또는 작업 완료 테스트
   - 기능 스텁(stub) 또는 가짜 문 테스트(fake door)
   - 기술 스파이크(technical spike)
   - 프로덕션에서의 A/B 테스트 (위험 완화와 함께)
   - 오즈의 마법사 접근법 (Wizard of Oz)
   - 설문 기반 검증 (의견이 아닌 행동 기반)

3. **따라야 할 핵심 원칙**:
   - 사용자의 의견이 아닌 실제 행동을 측정합니다
   - 사용자나 비즈니스를 위험에 빠뜨리지 않도록 책임감 있게 테스트합니다
   - 프로덕션 테스트(예: A/B 테스트)의 경우 위험 완화 전략을 설명합니다
   - 최소한의 노력으로 최대한의 검증된 학습을 목표로 합니다

4. **각 실험에 대해** 다음을 명시합니다:
   - **가정**: 우리가 믿는 것은 무엇인가?
   - **실험**: 이를 검증하기 위해 정확히 무엇을 할 것인가?
   - **지표**: 무엇을 측정할 것인가?
   - **성공 임계값**: 우리가 옳다면 기대되는 값은?

단계별로 생각합니다. 실험을 명확한 표 또는 구조화된 형식으로 제시합니다. 내용이 많으면 마크다운으로 저장합니다.

## 실험 설계 프레임워크

### Experiment Design Canvas

실험을 체계적으로 설계하기 위한 캔버스:

| 항목 | 내용 |
|------|------|
| **가설** | [검증할 가정을 명확한 문장으로] |
| **실험 유형** | 프로토타입/A/B 테스트/가짜 문/스파이크/오즈의 마법사 |
| **대상 지표** | [주요 지표 1개 + 보조 지표 1-2개] |
| **샘플 크기** | [필요한 최소 샘플 수] |
| **실험 기간** | [시작일 - 종료일] |
| **성공 기준** | [구체적 수치 — 실험 전 합의] |
| **실패 시 행동** | [피벗/재설계/기각] |
| **위험 완화** | [롤백 계획, 영향 범위 제한] |

활용 시점: 개별 실험을 상세 설계할 때 사용합니다. 가정 우선순위화(`prioritize-assumptions`) 직후, 실험 실행 전에 팀과 함께 캔버스를 채웁니다.

### A/B 테스트 계획 템플릿

프로덕션 A/B 테스트를 위한 구조화된 계획서:

| 항목 | 내용 |
|------|------|
| **테스트명** | [기능명_날짜] |
| **가설** | "[변경사항]을 적용하면 [지표]가 [X%] 개선될 것이다" |
| **대조군(Control)** | [현재 경험 설명] |
| **실험군(Variant)** | [변경된 경험 설명] |
| **주요 지표** | [전환율/클릭률/완료율 등 1개] |
| **가드레일 지표** | [악화되면 안 되는 지표] |
| **트래픽 배분** | [50/50 또는 90/10 등] |
| **최소 샘플 크기** | [통계 계산기로 산출] |
| **예상 기간** | [일 단위] |
| **의사결정 기준** | p < 0.05, 실질적 효과 크기 [X%] 이상 |

활용 시점: 프로덕션 환경에서 기능 변경의 효과를 측정할 때 사용합니다. 트래픽이 충분한 제품(일 활성 사용자 1,000명 이상)에서 효과적입니다.

### 통계적 유의성 가이드

A/B 테스트 결과 해석을 위한 핵심 개념:

- **p-value < 0.05**: 결과가 우연일 확률이 5% 미만. 업계 표준 임계값.
- **통계적 검정력 80%**: 실제 차이가 있을 때 이를 탐지할 확률. 샘플 크기에 비례.
- **최소 감지 효과(MDE)**: 탐지하고자 하는 최소 변화량. 작을수록 더 큰 샘플 필요.
- **실행 기간**: 최소 1-2 비즈니스 사이클(보통 2주). 요일별 변동을 포함해야 함.
- **다중 비교 보정**: 여러 지표를 동시에 테스트할 경우 Bonferroni 보정 적용.

### 실험 유형 분류

기존 제품에서 사용할 수 있는 실험 유형별 가이드:

| 실험 유형 | 적합한 상황 | 검증 속도 | 신뢰도 |
|-----------|------------|-----------|--------|
| **프로토타입 테스트** | UX/사용성 가정 검증 | 빠름 (1-3일) | 중간 |
| **가짜 문(Fake Door)** | 수요/관심도 검증 | 빠름 (1주) | 높음 |
| **기능 스텁(Feature Stub)** | 사용 의향 검증 | 빠름 (1주) | 높음 |
| **A/B 테스트** | 정량적 효과 검증 | 보통 (2-4주) | 매우 높음 |
| **기술 스파이크** | 기술 타당성 검증 | 보통 (1-2주) | 높음 |
| **오즈의 마법사** | 가치 가설 검증 (수동 실행) | 보통 (1-2주) | 높음 |
| **베타 출시** | 종합 검증 (제한된 사용자) | 느림 (4-8주) | 매우 높음 |

활용 시점: 검증할 가정의 유형에 따라 적합한 실험을 선택합니다. 사용성 → 프로토타입, 수요 → 가짜 문, 효과 → A/B 테스트 순으로 고려합니다.
