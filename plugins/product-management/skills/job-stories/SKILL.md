# Job Stories

'When [상황], I want to [동기], so I can [결과]' 형식으로 잡 스토리를 작성합니다. 사용자 상황과 결과에 초점을 맞춘 상세한 인수 기준이 포함된 스토리를 생성합니다.

**사용 시기:** 잡 스토리 작성, 사용자 상황과 동기 표현, JTBD 방식의 백로그 항목 생성, 또는 역할보다 사용자 맥락에 집중할 때.

**Arguments:**
- `$PRODUCT`: 제품 또는 시스템 이름
- `$FEATURE`: 잡 스토리로 분해할 신규 기능
- `$DESIGN`: 디자인 파일 링크 (Figma, Miro 등)
- `$CONTEXT`: 사용자 상황 또는 잡 시나리오

## 단계별 프로세스

1. **필요를 촉발하는 사용자 상황 파악**
2. **사용자 행동의 근본적인 동기 정의**
3. **사용자가 달성하려는 결과 명확화**
4. **JTBD 프레임워크 적용:** 역할이 아닌 잡(job)에 집중
5. **결과 달성 여부를 검증하는 인수 기준 작성**
6. **관찰 가능하고 측정 가능한 언어 사용**
7. **디자인 목업 또는 프로토타입 링크 연결**
8. **상세 인수 기준이 포함된 잡 스토리 출력**

## 스토리 템플릿

**제목:** [잡의 결과 또는 성과]

**설명:** When [상황], I want to [동기], so I can [결과].

**디자인:** [디자인 파일 링크]

**인수 기준:**
1. [상황이 올바르게 인식됨]
2. [시스템이 원하는 동기를 지원함]
3. [진행 상황 또는 피드백이 표시됨]
4. [결과가 효율적으로 달성됨]
5. [엣지 케이스가 우아하게 처리됨]
6. [연동 및 알림이 작동함]

## 잡 스토리 예시

**제목:** 주간 간식 지출 추적

**설명:** When I'm preparing my weekly allowance for snacks (상황: 주간 간식 비용을 준비할 때), I want to quickly see how much I've spent so far (동기: 지금까지 얼마나 썼는지 빠르게 확인하고 싶음), so I can make sure I don't run out of money before the weekend (결과: 주말 전에 돈이 떨어지지 않도록).

**디자인:** [Figma 링크]

**인수 기준:**
1. '주간 지출 현황' 섹션과 함께 지출 요약 표시
2. 지출 항목 등록 시 실시간 업데이트
3. 주간 예산의 0~100%를 나타내는 진행률 표시줄
4. 눈에 잘 띄는 색상으로 잔여 예산 강조 표시
5. 카테고리별 세부 지출 내역
6. 예산의 80% 도달 시 알림 발송
7. 목요일 저녁 예산의 90% 도달 시 주말 관련 알림
8. 세부 내역으로의 간편한 접근 및 탐색

## 산출물

- 기능에 대한 완전한 잡 스토리 세트
- 각 스토리는 'When...I want...so I can' 형식을 따름
- 결과에 초점을 맞춘 6~8개의 인수 기준
- 사용자 상황과 동기를 강조한 스토리
- 디자인 및 프로토타입으로의 명확한 링크

### Job Story vs User Story 비교표

**활용 시점:** 프로젝트에 적합한 스토리 형식을 선택하거나 팀에 차이를 설명할 때.

| 구분 | User Story | Job Story |
|------|-----------|-----------|
| 형식 | As a [역할], I want [행동], so that [혜택] | When [상황], I want to [동기], so I can [결과] |
| 초점 | 사용자 역할 중심 | 사용자 상황/맥락 중심 |
| 장점 | 역할별 요구사항 정리에 유리 | 실제 사용 맥락 포착에 유리 |
| 약점 | 역할이 모호하면 스토리도 모호 | 상황 정의가 어려울 수 있음 |
| 적합 상황 | 역할이 명확한 B2B/엔터프라이즈 | 다양한 맥락의 B2C/모바일 |

### 상황-동기-결과 구조 가이드

**활용 시점:** 잡 스토리의 각 구성 요소를 정확하게 작성할 때.

| 구성 요소 | 작성 원칙 | 질문 | 예시 |
|----------|----------|------|------|
| When [상황] | 구체적 맥락과 트리거 | 사용자가 언제, 어디서, 왜 이 필요를 느끼는가? | "주간 예산을 확인하려고 앱을 열었을 때" |
| I want to [동기] | 근본적 욕구, 기능이 아닌 목적 | 사용자가 진정으로 원하는 것은? | "남은 예산을 한눈에 파악하고 싶다" |
| So I can [결과] | 측정 가능한 성과 | 이것이 달성되면 무엇이 달라지는가? | "주말까지 예산 내에서 소비할 수 있다" |

### 잡 스토리 품질 체크리스트

**활용 시점:** 작성된 잡 스토리의 완성도를 검증할 때.

| 기준 | 확인 질문 | 통과 기준 |
|------|----------|----------|
| 상황 구체성 | 상황이 특정 맥락을 포함하는가? | "앱을 열 때"가 아닌 "출퇴근 중 잔액 확인 시" |
| 동기 진정성 | 기능이 아닌 근본 욕구인가? | "버튼 클릭"이 아닌 "빠르게 파악" |
| 결과 측정성 | 결과 달성을 확인할 수 있는가? | 관찰 가능한 행동 또는 상태 변화 |
| 솔루션 독립성 | 특정 UI/기술에 종속되지 않는가? | 다양한 구현 방식이 가능 |
| 인수 기준 연결 | 결과를 검증하는 기준이 있는가? | 최소 4개 이상의 인수 기준 |

### 작성 예시: 좋은 예 vs 나쁜 예

**활용 시점:** 팀원에게 잡 스토리 작성법을 교육하거나 리뷰할 때.

| 구분 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| 상황 | When I use the app | When I'm commuting and want to check today's schedule |
| 동기 | I want to click the button | I want to see today's meetings at a glance |
| 결과 | So I can use the feature | So I can prepare talking points before arriving |
| 전체 | When I use the dashboard, I want to see data, so I can use it | When I'm preparing for a weekly review, I want to see trend changes at a glance, so I can focus discussion on areas needing attention |

### 안티패턴

**활용 시점:** 잡 스토리 리뷰 시 흔한 실수를 방지할 때.

- **솔루션 주입**: 상황에 특정 UI 요소를 포함 → 상황은 사용자 맥락만 기술
- **모호한 동기**: "쉽게 하고 싶다" → 무엇을 쉽게? 구체적 행동 명시
- **측정 불가 결과**: "더 나은 경험" → 어떤 상태가 달라지는지 명시
- **역할 혼입**: "관리자로서" → 역할이 아닌 상황으로 전환
- **과대 범위**: 하나의 스토리에 여러 잡 포함 → 잡당 하나의 스토리로 분리
