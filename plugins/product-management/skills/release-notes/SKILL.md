## 릴리스 노트 생성기

기술적인 티켓, PRD, 또는 내부 변경 이력을 사용자 대상의 완성도 높은 릴리스 노트로 변환합니다.

### 맥락

**$ARGUMENTS**에 대한 릴리스 노트를 작성합니다.

사용자가 파일(JIRA 내보내기, Linear 티켓, PRD, Git 로그, 또는 내부 변경 이력)을 제공하면 먼저 읽으세요. 제품 URL을 언급한다면 웹 검색을 활용하여 제품과 대상 사용자를 파악하세요.

### 지시사항

1. **원본 자료 수집**: 제공된 모든 티켓, 변경 이력, 또는 설명을 읽으세요. 다음 내용을 추출하세요:
   - 변경된 내용 (기능, 개선 사항, 또는 버그 수정)
   - 영향받는 대상 (어떤 사용자 세그먼트)
   - 중요한 이유 (사용자 혜택)

2. **변경 사항 분류**:
   - **신규 기능**: 완전히 새로운 기능
   - **개선 사항**: 기존 기능 향상
   - **버그 수정**: 해결된 문제
   - **호환성 변경**: 사용자 조치가 필요한 내용 (마이그레이션, API 변경)
   - **지원 중단**: 서비스가 종료되는 기능

3. **각 항목 작성** 시 다음 원칙을 따르세요:
   - 기술적 변경이 아닌 사용자 혜택을 먼저 제시
   - 평이한 언어 사용 — 전문 용어, 내부 코드명, 티켓 번호 지양
   - 각 항목은 1~3문장으로 유지
   - 사용자가 이미지나 스크린샷을 제공하면 포함

   **변환 예시**:
   - 기술적 표현: "Implemented Redis caching layer for dashboard API endpoints"
   - 사용자 대상: "대시보드가 최대 3배 빠르게 로드되어, 기다리는 시간보다 분석하는 시간이 늘어납니다."

   - 기술적 표현: "Fixed race condition in concurrent checkout flow"
   - 사용자 대상: "트래픽이 많은 시간대에 일부 주문이 실패하는 문제를 수정했습니다."

4. **릴리스 노트 구조화**:

   ```
   # [제품명] — [버전 / 날짜]

   ## 신규 기능
   - **[기능명]**: [기능이 하는 일과 중요한 이유를 1~2문장으로 설명]

   ## 개선 사항
   - **[영역]**: [무엇이 개선되었고 어떻게 도움이 되는지]

   ## 버그 수정
   - [사용자 관점에서 설명한 문제 해결 내용]

   ## 호환성 변경 (해당하는 경우)
   - **조치 필요**: [사용자가 해야 할 일]
   ```

5. **톤 조정**: 제품의 보이스에 맞게 — B2B는 전문적으로, 소비자용은 친근하게, API는 개발자 중심으로.

마크다운 문서로 저장하세요. 사용자가 HTML 또는 다른 형식을 원하면 그에 맞게 변환하세요.

## 릴리스 유형별 가이드

| 유형 | 범위 | 예시 | 작성 수준 |
|------|------|------|----------|
| **Major (v2.0)** | 핵심 기능 추가/변경, UX 대폭 개편 | 대시보드 전면 리디자인, 새 플랫폼 지원 | 상세 설명 + 스크린샷 + 마이그레이션 가이드 |
| **Minor (v1.3)** | 기존 기능 개선, 새 하위 기능 | 필터 옵션 추가, 성능 개선 | 간결한 혜택 중심 설명 |
| **Patch (v1.3.1)** | 버그 수정, 보안 패치 | 로그인 오류 수정, 보안 취약점 해결 | 1-2문장 요약 |

활용 시점: 릴리스 규모에 따라 작성 깊이와 커뮤니케이션 채널을 결정할 때. Major는 블로그 + 이메일, Minor는 인앱 공지, Patch는 변경 로그만.

> **토킹 포인트**: [고객 문의 시 핵심 메시지]

> **필요 조치**: [고객 측 조치 사항]
> **지원**: [문의처 및 문서 링크]

## Changelog vs Announcement 가이드

| 구분 | Changelog | Announcement |
|------|-----------|-------------|
| **목적** | 모든 변경 사항의 기록 보관 | 주요 업데이트 홍보 및 알림 |
| **대상** | 개발자, 파워 유저 | 전체 사용자, 잠재 고객 |
| **톤** | 사실적, 기술적 | 혜택 중심, 마케팅적 |
| **빈도** | 모든 배포마다 | Major/중요 Minor만 |
| **형식** | 날짜별 변경 목록 | 블로그, 이메일, 인앱 배너 |
| **포함 내용** | 기능 추가, 변경, 수정, 제거 전부 | 핵심 기능 + 혜택 + CTA |

활용 시점: Changelog는 투명성과 기술 문서 목적, Announcement는 마케팅과 사용자 참여 목적. 두 가지를 혼동하면 기술 정보가 부족하거나 마케팅 효과가 떨어진다.

## 작성 체크리스트

릴리스 노트 발행 전 최종 검토:

- [ ] 모든 변경 사항이 사용자 혜택 관점으로 서술되었는가?
- [ ] 호환성 변경(Breaking Changes)이 명확히 표시되었는가?
- [ ] 전문 용어, 내부 코드명, 티켓 번호가 제거되었는가?
- [ ] 대상 독자에 맞는 톤이 적용되었는가?
- [ ] 필요한 스크린샷/GIF가 포함되었는가?
- [ ] 마이그레이션 가이드 또는 조치 사항이 작성되었는가? (해당 시)
- [ ] 지원/문의 링크가 포함되었는가?
- [ ] 이해관계자(CS, 영업, 마케팅) 사전 공유가 완료되었는가?
