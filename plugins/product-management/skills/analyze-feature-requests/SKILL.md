## 기능 요청 분석

제품 목표 대비 고객 기능 요청을 분류, 평가, 우선순위화합니다.

### 맥락

**$ARGUMENTS**에 대한 기능 요청을 분석합니다.

사용자가 파일(스프레드시트, CSV, 또는 기능 요청이 담긴 문서)을 제공하면 먼저 읽고 분석합니다. 데이터가 구조화된 형식이라면 요약 테이블 작성을 고려합니다.

### 도메인 맥락

고객이 솔루션을 설계하도록 허용하지 않습니다. **기능이 아닌 기회(문제)**를 우선시합니다. **기회 점수(Opportunity Score)**(Dan Olsen)를 사용하여 고객이 보고한 문제를 평가합니다: 기회 점수 = 중요도 × (1 − 만족도), 0–1로 정규화. 전체 세부 내용과 템플릿은 `prioritization-frameworks` 스킬을 참고합니다.

### 지시 사항

사용자가 제품 목표를 설명하고 기능 요청을 제공합니다. 다음 단계를 거칩니다:

1. **목표 파악**: 우선순위 결정을 안내할 제품 목표와 원하는 결과를 확인합니다.

2. **요청을 테마로 분류**: 관련 요청을 그룹화하고 각 테마에 이름을 붙입니다.

3. **전략적 정렬 평가**: 각 테마가 명시된 목표와 얼마나 잘 맞는지 평가합니다.

4. **상위 3개 기능 우선순위 결정** 기준:
   - **영향**: 고객 가치와 영향받는 사용자 수
   - **노력**: 필요한 개발 및 디자인 리소스
   - **위험**: 기술적 및 시장 불확실성
   - **전략적 정렬**: 제품 비전 및 목표와의 적합성

5. **각 상위 기능에 대해** 다음을 제공합니다:
   - 근거 (고객 니즈, 전략적 정렬)
   - 고려할 대안적 솔루션
   - 고위험 가정
   - 최소한의 노력으로 해당 가정을 테스트하는 방법

단계별로 생각합니다. 마크다운으로 저장하거나 구조화된 출력 문서를 작성합니다.

## 기능 요청 분석 프레임워크

### Feature Request Clustering (요청 클러스터링)

대량의 기능 요청을 체계적으로 그룹화하는 방법:

1. **원시 데이터 수집**: 모든 채널(CS 티켓, 설문, 영업 피드백, 커뮤니티)의 요청을 한 곳에 모읍니다.
2. **문제 기반 재정의**: 요청된 솔루션이 아닌 근본 문제로 재정의합니다. "다크 모드 추가해주세요" → "장시간 사용 시 눈의 피로 문제".
3. **테마 클러스터링**: 유사한 문제를 테마로 묶습니다. 테마당 3-15개 요청이 적절합니다.
4. **빈도-긴급도 태깅**: 각 클러스터의 요청 빈도와 긴급도를 집계합니다.

활용 시점: 미분류 기능 요청이 20개 이상 누적되었을 때 사용합니다. 분기별 백로그 정리 또는 로드맵 계획 전에 실행하면 효과적입니다.

### Kano 모델 분류

기능 요청을 고객 만족도 영향 기준으로 분류합니다:

| 카테고리 | 설명 | 전략 |
|----------|------|------|
| **필수(Must-be)** | 없으면 불만, 있어도 당연시 | 반드시 구현 — 경쟁 테이블 스테이크 |
| **일원적(One-dimensional)** | 잘 할수록 만족도 비례 상승 | 경쟁사 대비 수준 평가 후 결정 |
| **매력적(Attractive)** | 없어도 불만 없지만, 있으면 감동 | 차별화 기회 — 선택적 투자 |
| **무관심(Indifferent)** | 있든 없든 만족도 변화 없음 | 구현하지 않음 — 리소스 낭비 |
| **역효과(Reverse)** | 오히려 만족도를 떨어뜨림 | 즉시 제거 또는 옵션화 |

활용 시점: 기능 요청 클러스터링 후, 각 테마의 전략적 가치를 평가할 때 사용합니다. 고객 설문(Kano 설문 기법)과 병행하면 정확도가 높아집니다.

### 분석 프로세스

전체 기능 요청 분석의 권장 순서:

1. **수집** → 모든 채널의 기능 요청을 통합 (1-2일)
2. **클러스터링** → 문제 기반 테마로 그룹화 (반나절)
3. **Kano 분류** → 각 테마의 만족도 영향 분류 (반나절)
4. **수요 분석** → 빈도, 긴급도, 매출 영향 정량화 (1일)
5. **전략 정렬** → 제품 비전 및 분기 목표와 매핑 (반나절)
6. **우선순위 결정** → 상위 3-5개 기능 선정 및 근거 문서화 (반나절)

활용 시점: 분기별 로드맵 계획 전에 전체 프로세스를 실행합니다. 긴급 요청은 별도 트리아지 프로세스로 처리합니다.
